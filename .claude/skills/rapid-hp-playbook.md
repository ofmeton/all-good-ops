# Rapid HP Playbook（BSA事業 標準運用手順）

> **ステータス: 承認済（2026-04-23）**
> 起案日: 2026-04-22 / SSOT: `wiki/business/bsa/pricing-catalog.md`

## 概要

BSA 事業（工藤陸名義 / HP・LP制作 × 超短納期 × 広告運用）の**受注〜納品〜継続運用までの標準手順**をまとめたプレイブック。`rapid-hp-operator` エージェントが主に参照し、`system-engineer` が実装時に併用する。

## 対象

- **メイン**: rapid-hp-operator
- **副次**: system-engineer（制作時）、freelance-scout（案件評価時）

## 前提の SSOT

以下は **必ず** `pricing-catalog.md` を正とする:
- 価格・納期・SLA
- AI表記ルール（「AI活用」のみ、固有名詞禁止）
- 名義ルール（工藤陸）
- portfolio URL / メールアドレス

## 1. 提案フェーズのプレイブック

### 1.1 案件評価 3ステップ

1. **fit_score 算出**（`outputs/bsa/watcher-design.md` §2 の配点表）
2. **除外条件チェック**:
   - 認定ランサー限定 / 実績10件以上必須
   - 薬機法・景表法リスク大
   - 情報商材・MLM・副業紹介
   - 予算1万円未満
3. **商品ライン判定**:
   - LP単発希望 → T1 (L1 ¥30k)
   - コーポレート希望 → T2 (L2 ¥80k)
   - LP + 広告運用 → T3 (L3 ¥100k)
   - 既存サイト修正 → T4 (L4 ¥10k〜)
   - 広告単発希望 → T5

### 1.2 提案文の必須3要素（B-C層差別化の核）

需給分析§2.5 で判明した事実: **B-C層の応募文は8割がテンプレ丸写し**。以下3要素を入れるだけで上位10%に入る:

1. **事業理解の要約**（冒頭3点の1つに固定）
   - 発注者の事業課題を1-2文で要約
   - 例: "〇〇様は地域密着型の注文住宅ビルダーで、若年層の認知獲得に課題をお持ちと推察します"

2. **数値KPIの見立て**（必ず1つ入れる）
   - 想定CVR / CPA / 月間CV数などの仮説
   - 例: "想定CVR 2-3%、月広告費30万円で問い合わせ30-45件の獲得が現実的"

3. **広告運用への接続文**（中盤に配置）
   - L3 系でなくても触れる
   - 例: "LP公開後の広告運用も含めてご相談可能です（月額〇〇円〜）"

### 1.3 業種別ヒアリング質問集

**提案段階で "事業理解の要約" を書くための調査質問**。受注前に発注者へ聞けるものは聞き、聞けないものは案件詳細から推測する。

#### 建築・不動産系

- 注文住宅 / 建売 / リフォーム / 土地 / 賃貸 の軸
- 商圏（市町村単位）
- ターゲット年齢層（20-30代 / 40-50代 / シニア）
- 競合のHP
- 現状の集客方法（Web / 折込 / 紹介 / SUUMO等ポータル）
- 月間問い合わせ目標数
- 完成事例の写真・素材準備状況

#### 士業（社労士・行政書士・税理士）

- 専門領域（助成金 / 相続 / 開業 / 労務 等）
- 事業規模のターゲット（個人 / SOHO / 中小法人 / 100名以上）
- 無料相談の有無と時間
- 商圏（市区 or 全国）
- 現状の問い合わせ経路
- 実績数値（顧問先数 / 相談件数）
- スタッフ数と顔出し可否

#### 医療（自費診療・クリニック）

- 診療科と専門領域
- 自費の主要メニュー（インプラント / 矯正 / ICL / ヒアルロン酸 等）
- 薬機法・医療広告ガイドライン遵守の現状確認
- 院長経歴・症例数・使用機器
- Before/After 素材の有無と薬機法上の扱い可否
- 予約システムの有無（LINE / Web予約 / 電話）
- 月間来院数目標

#### BtoB SaaS・サービス

- サービス概要と価格帯
- 導入企業の規模感
- 営業フロー（資料請求→商談→受注 の歩留まり）
- 既存の資料・ケーススタディ
- LP の CV ポイント（資料請求 / 無料トライアル / 問い合わせ）

## 2. 受注後フェーズのプレイブック

### 2.1 30分ヒアリングのアジェンダ雛形

| 時間 | 項目 | ゴール |
|---|---|---|
| 0-3分 | アイスブレイク + 本日の流れ説明 | 発注者の緊張を解く |
| 3-11分 | 事業概要とターゲット顧客 | ペルソナ像を3行で言語化 |
| 11-19分 | 成功イメージと KPI | 数値ゴールを確定（or 仮置き） |
| 19-24分 | デザイン方向性（参考サイト2-3個） | トンマナの方向を合意 |
| 24-27分 | 素材・原稿の準備状況 | 準備済/未準備の仕分け |
| 27-30分 | 納期・公開タイミング・契約条件 | SLA カウント開始日を合意 |

### 2.3 SLA タイマー管理

| 商品 | SLA | 中間レビュー |
|---|---|---|
| L1 | 72h | 48h時点（残24h） |
| L2 | 7日 | 3日目 + 5日目 |
| L3 | 96h + 翌日配信 | 48h時点（残48h） |
| L4 | 24h着手 / 3日以内完了 | 24h 経過時点で進捗確認 |

**警戒水準**:
- 緑: 残75%以上 / 想定通り
- 黄: 残50%以下で進捗30%未満 → ユーザーにアラート
- 赤: 残25%以下で進捗50%未満 → クライアントへ "超過可能性" を先出しする判断

## 5. 継続運用フェーズのプレイブック

### 5.1 L3 初月終了時の動き

- **Week 4 頃**（初月終了 1週間前）: 継続運用オファーの準備着手
- **Week 4 末**: 月次レポート PDF + 継続運用オファーを送付

### 5.2 L3-C 運用の優先順序（pricing-catalog §1 L3-C 準拠）

| フェーズ | 期間 | やること | LPに触るか |
|---|---|---|---|
| Month 1 | Week 1-4 | Week1: 学習期間 / W2: 検索クエリ整理 / W3-4: 入札戦略調整 | **触らない** |
| Month 2 | Week 5-8 | オーディエンス/時間帯/地域/デバイス最適化・広告表示オプション拡充 | **触らない** |
| Month 3+ | — | ヒートマップ分析・FV/CTA/フォーム改善・派生LP検証 | **触る（別料金）** |

### 5.3 月次レポートの最小構成

1. 今月の主要指標（CPA / CVR / CV数 / 広告費 / ROAS）
2. 前月比の変化と要因仮説
3. 今月実施した施策
4. 来月の計画
5. 次月の広告費案（据え置き / 増額 / 減額）

### 5.4 解約兆候の検知

- 月次MTG で成果への不満発言
- 広告費減額の要請2ヶ月連続
- 月次レポート未読2ヶ月連続
- 返信遅延が平均3日超に

→ 検知時は `rapid-hp-operator` からエスカレーション。改善提案 or 解約交渉準備。

## 6. 顧客向け表現ルール（重要 / pricing-catalog §1 L3-C 末尾より）

- **NG**: 「配信直後にLPをいじるのは運用として雑」「早すぎる」「筋が悪い」
- **OK**:
  - 「広告側から順番に打ちきる運用設計」
  - 「Month 1-2 は広告側の最適化に集中し、必要に応じて LP へ」
  - 「機械学習の学習期間を確保しつつ、打てる手を順番に打ちきります」

**理由**: 否定・防御的な表現は顧客に不必要な警戒を与える。内部の運用指針はそのまま顧客に見せない。
