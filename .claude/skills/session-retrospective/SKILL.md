# セッション振り返り（Session Retrospective）

## 概要

ユーザーがセッションの作業・やり取りを振り返りたい時に起動する。**前回提案の再計測 →**
気づき抽出 → 反映先候補の提示 → 人間確認 → 実装までを標準化し、一度きりの反省で
終わらせず次回以降の運用へ確実に反映させる。

- **いつ**: ユーザーが「振り返って」「セッション振り返り」等と明示要求した時のみ。**自動発火しない**
- **何のために**: memory / improvement-log / CLAUDE.md / エージェント定義 / wiki / 新規スキルへ反映

## 中核原則（最優先・他の手順より上位）

1. **ループを閉じる**: 振り返りは「前回提案の再計測」から始める（§0.5）。書きっぱなしにしない
2. **保存はデフォルト「追記 or 保留」**: 新規 memory/skill は§6の保存関門を通った時だけ作る（肥大防止）
3. **本文は中身がある節だけ出す**: 固定テンプレを埋めるための水増し・創作をしない
4. **自己採点は付けない**（定量化せず定性的な気づきのみ）。迎合的な褒めも禁止

## 標準構造

### 0. 事実情報の raw 保存漏れチェック（必ず最初）

CLAUDE.md §事実情報の自動 raw 保存ルールの遵守確認。**振り返り本体より先に**実施し、漏れはその場で保存。

1. セッション内のユーザー発話を走査し、people / contracts / situations / misc の事実を列挙
2. 対応する `raw/facts/<カテゴリ>/YYYY-MM-DD-<slug>.md` が当日作成済みか確認（`ls raw/facts/<cat>/ | grep <date>`）
3. 未保存はその場で保存・1行通知（人間確認不要）。漏れた理由を §2 表に必ず記録
4. 対象外: 雑談 / 既存情報の再言及 / 質問・依頼そのもの / 仮説・推測

### 0.5 前回フォローアップ（ループの再計測・必須）

前回までの improvement-log を読み返し、提案が今回守れたかを照合する。**これがこのスキルの実働化の要**（書きっぱなし＝儀式化を防ぐ）。

1. `tail -n 20 data/improvement-log.jsonl` で直近の retro 由来エントリを確認
2. 各提案の `status` を更新（フィールド無ければ付与）: `applied`=今回実践できた / `verified`=効果も確認 / `open`=未着手 or 再発
3. **2回連続で open**＝定着していない兆候。§4 で構造的な根本原因を必ず掘る
4. 結果を 1-2 行で表示（例「前回 3 件中 2 applied / 1 再発」）

※ improvement-log の自動集計（`self-improve.sh` 週次・`monthly-audit.sh` 月次）は cron 全停止中（memory `project_cron_automation_disabled.md`）。**当面この読み返しが唯一の消費経路**。

### 1. 良かった点（3-5項目・短く）
意図的にうまく動けた点を具体行動と理由セットで。「全体的に良かった」等の迎合は禁止。

### 2. 詰まった瞬間・二度手間（表形式）

| # | 事象 | 原因 | 先回りできたポイント | 本来すべき動き |
|---|---|---|---|---|

- 「詰まった」=手戻り・失敗・想定外の確認プロンプト / 「二度手間」=同操作・同確認の2回以上反復
- 原因は表層症状でなく構造的要因まで掘る

### 3. 自動化・効率化の余地
3回以上の反復操作・型化できる判断・permissions 追加で減らせる許可プロンプト・ルーティング改善。**スキル化候補は§6保存関門を通す**。

### 4. 次回への改善提案
**アクション可能な粒度で**。「気をつける」「注意する」は検証不能ゆえ禁止。
例:「次回 `.zshrc` 編集時は現セッションに反映されない旨を即明示する」。

### 5. 観点レンズ（ヒットした時だけ・無ければ節ごと省略）

水増し防止のため**該当があった時のみ**出す。Claude 自身の行動を振り返って抽出する:

- 🔧 **未活用資産**: 使えたのに使わなかった repo 資産（skill / agent / wiki / memory / script）。「次回はこれを使う」までセット
- 💬 **プロンプト改善**: 「こう指示してくれたら一発で動けた」をユーザーへ 1-2 点
- ⚡ **Claude 機能の出番**: 使うべきだった機能（worktree / Workflow / parallel agents / plan mode / ToolSearch 等）
- 🪙 **トークンコスパ**: より安く済んだ手（全文 Read→offset/limit、サブエージェント分散、MCP より CLI 等）

### 6. 反映先候補（保存関門 → SAFE/RISKY 分類）

#### 保存関門（候補を出す前に必ず通す・作る方向のバイアスを相殺）

デフォルトは「既存に追記」or「保留」。新規作成は関門通過時のみ:

- **memory**: 既存 `feedback_*` に1行追記で代替不可か？ recall されて判断が変わる新規事実か？ → No なら新規作成しない。最終関門=**MEMORY.md 索引を1行増やすに値するか**（索引は毎セッション全文ロード＝コスト中心）
- **wiki 優先**: 一般化・連結できる原則は memory でなく `wiki/self/engineering-principles.md` へ（memory=atomic recall / wiki=連結・高次化）
- **スキル化**: 既存 skill/agent で代替不可か？ 今後30日で**2回以上発火する具体シーンを書ける**か？ → 書けなければ improvement-log に `status=open` で寝かせ、即スキル化しない

#### 反映先カテゴリ

| カテゴリ | 反映先 | 何を書くか |
|---|---|---|
| memory (feedback) | `~/.claude/projects/.../memory/feedback_*.md` | ユーザーの好み・作業スタイルの発見（既存追記優先） |
| wiki | `wiki/self/engineering-principles.md` 等 | 一般化・連結できる原則 |
| improvement-log | `data/improvement-log.jsonl` | プロセス失敗・保留中のスキル化候補・次回監視指標（`status` 付き） |
| CLAUDE.md | `CLAUDE.md` | ルーティングの盲点・MCP/人間確認ルールの穴 |
| エージェント定義 | `.claude/agents/**/*.md` | ツール権限不足・参照スキル漏れ・手順明示不足 |
| 新規スキル化候補 | `.claude/skills/` | 関門を通った反復手順・型化できる判定 |

#### SAFE（まとめ承認可）
memory/wiki への追記・improvement-log 追記・CLAUDE.md やエージェント定義の文言/参照/権限の1行調整。

#### RISKY（1件ずつ承認）
CLAUDE.md のルーティング表大幅変更・戦略 KGI・人間確認ルール変更 / エージェントの新規追加・削除・統合 / permissions 変更 / **新規スキルファイルの確定追加** / 複数ファイル横断の構造変更。

## 実行フロー（順序）

1. **スコープ把握** — 既定は直近セッション全体。フェーズ指定があればそこだけ。過去横断は `monthly-audit.sh` の領域
2. **§0 raw 保存漏れチェック** → 未保存はその場で補完
3. **§0.5 前回フォローアップ** → improvement-log 読み返し・status 更新・再計測表示
4. **§1-5 を対話内に表示**（中身がある節だけ。この段階では本体を書き込まない）
5. **§6 反映先候補** → 保存関門を通してから SAFE/RISKY 分類で提示・承認取得
6. **承認後に反映** → 本体を `outputs/retrospectives/YYYY-MM-DD-HHmm[-topic].md` に保存（topic は lowercase-hyphen、冒頭に日時+対象サマリ）
7. **`wiki/hot.md` 更新（必須）** — Last Updated / Current Focus / Recently Touched(直近7件) / Open Questions。500 words 以内。詳細は `wiki/SCHEMA.md` §ホットキャッシュ

## 絶対にやらないこと

1. **人間確認なしに memory / CLAUDE.md / エージェント定義 / permissions を書き換える**
2. **保存関門を通さず memory / 新規スキルを作る**（既存追記・保留を検討しない）
3. **§0.5 前回フォローアップを飛ばす**（書きっぱなし＝儀式化を招く）
4. **中身のない節を水増しする** / **迎合的な振り返り**を出す
5. **具体性のない改善提案**（「次は注意します」）を出す
6. **振り返りをファイル化せず対話内だけで終える** / **反映候補を出さずに終える**
7. **ユーザーに自己採点を求める**
