# エージェント新設プロトコル

> 目的: 新規エージェント追加時の重複防止・品質確保・体制整合を一貫した手順で行う。CLAUDE.md「エージェント管理パイプライン 2. 取り込み」の実務手順書。

## 発動タイミング
- ユーザーから「〇〇エージェントを追加して」の依頼
- usage-log 分析で新規候補が検出された時（org-designer 起動時）
- 既存エージェントでは捌けない業務領域が見つかった時

## 人間確認
- 新規エージェント追加は **必ず人間承認が必要**（CLAUDE.md 人間確認ルール「エージェント変更」カテゴリ）
- 本プロトコル開始時に「〇〇を新設しますがよろしいですか？」で承認取得

## 手順

### Step 1: 既存確認（最重要）

**会話冒頭の snapshot git status は信用しない。** 必ず最新を取り直す。

```bash
# 既存エージェント一覧
find .claude/agents -name "*.md"

# 既存スキル一覧
ls .claude/skills/*.md

# キーワードで類似役割を検索（例: BSA関連なら "bsa" "rapid" "hp"）
grep -rl "<キーワード>" .claude/agents/ .claude/skills/ 2>/dev/null

# 最新の git 状況（untracked も含める）
git status
```

類似役割が見つかったら、ユーザーに以下の選択肢を提示：
- **採用**（既存を使う）
- **拡張**（既存を改修）
- **新設**（明確に役割が違う場合のみ）

**根拠**: 2026-04-23 の事故で、既に承認済みだった `rapid-hp-operator` の存在を見落として `bsa-operator` を重複新設してしまった。会話冒頭の git status は古い snapshot のため、untracked ファイルが反映されていなかった。

### Step 2: 役割定義

以下をユーザーと合意してから実装に進む：
- 役割の定義（1-2文で）
- 守備範囲（3-5項目）
- 非守備範囲（既存エージェントとの境界）
- 受け取るべき依頼の特徴（例示3つ以上）
- トリガーキーワード（ルーティング表に入る文言）

### Step 3: 定義ファイル作成

既存エージェント（例: `shopify-operator.md`）のフォーマットに準拠。最小セクション：

- 役割の定義
- 守備範囲 / 非守備範囲
- 受け取るべき依頼の特徴
- 起動時に必ず行うこと（関連 context/SSOT/前提ファイル読み込み）
- 出力の品質基準
- 参照すべきスキル（必須 / 参考）
- 他エージェントとの連携ルール
- escalation 条件
- 人間確認が必要な条件
- 使ってよい / 慎重に使うべきツール
- トーン / スタイル
- 成果評価の観点
- よくある失敗
- 引き継ぎフォーマット

配置: `.claude/agents/<部門>/<エージェント名>.md`

### Step 4: CLAUDE.md の3点同期

以下を同時に更新（漏れなし）：
1. **ルーティング表**（キーワード → 担当エージェント）
2. **部門一覧**の該当セクション
3. **「部門一覧」ヘッダーのエージェント総数**（例: 33→34）

※ CLAUDE.md への複数 Edit は、同一ファイルへの並列 Edit が衝突しやすい。1 Edit ずつ送るか、Write で全文書き直すほうが安全。

### Step 5: 関連スキルの検討

- 新エージェントが参照すべき手順書をスキルとして切り出す必要があるか
- 既存スキルで代替できるか
- 必要なら `.claude/skills/` に追加し、CLAUDE.md スキル一覧・冊数も更新

### Step 6: 初期ランク設定

- `data/agent-ranks.json` に新エージェントを追加（初期ランク = N）
- 月次品質監査（monthly-audit.sh）でスコアリング対象に入る

### Step 7: git commit

- 定義ファイル + CLAUDE.md 更新 + ranks 更新を**1コミット**にまとめる
- メッセージ例: 「〇〇エージェント新設: 役割と守備範囲の定義」

## 完了チェックリスト

- [ ] Step 2 役割定義: ユーザーと合意済み
- [ ] Step 3 定義ファイル作成: フォーマット準拠
- [ ] Step 4-1 ルーティング表に追加
- [ ] Step 4-2 部門一覧に追加
- [ ] Step 4-3 エージェント総数更新

## よくある失敗
- 会話冒頭の snapshot git status を信用して untracked エージェントを見落とす
- エージェント総数の更新漏れ（ルーティング表だけ更新して部門一覧ヘッダーの数字を忘れる）
- 既存エージェントとの境界が曖昧なまま走り出す → あとで整理コスト発生
- 関連スキルを同時に作らず、エージェント単独で動かして詰まる
- 同一ファイルへの並列 Edit で衝突エラーを連発する
