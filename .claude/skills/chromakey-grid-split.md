# Chroma-key グリッド切り出し（Chromakey Grid Split）

## 概要

マゼンタ/シアン等のフラット背景上にグリッド配置された複数のデザイン素材を、1ファイル=1素材のRGBA透過PNGに分割する。LP/HP制作で生成AIから貰う「素材シート」の処理に使う。

- **誰が**: system-engineer / rapid-hp-operator
- **いつ**: 「グリッド画像を切り出して透過化」「素材シートを分解」「カテゴリ別フォルダに分けて」等の依頼
- **何のために**: 1枚の素材シートPNGを実装で使える単位の透過PNGに分解する

## トリガー（自然文例）

- 「このフォルダ内のグリッド画像を切り出して透過にして」
- 「素材シートを 1カテゴリ1フォルダで保存して」
- 「画像をグリッドで分割して背景透過」

## 前提条件

- **共通venv**: `~/.venvs/img-tools/`（numpy / Pillow / scipy 常設）
- **テンプレ**: `outputs/templates/chromakey_grid_split.py` をコピーして使う
- **背景**: マゼンタ系（#FF00FF系）またはシアン系（#00FFFF系）のフラット色
- **依頼者は通常、各シートの「行x列の構成」を把握している**（不明なら原画像を読み込んで確認）

## 標準フロー

### Step 0 — 着手前の構造把握（**手戻り防止に必須**）

1. 原画像をすべて Read で読み込み目視
2. 各画像について以下をメモする
   - グリッド構成（行×列）
   - 背景色系統（マゼンタ/シアン）
   - 不規則レイアウトかどうか（要素サイズや位置がバラバラなら irregular）
3. **bg比率プロファイル走査で構造を確認**（任意・不安な時のみ）

```python
# 各行/列の bg ratio を見る（手早く構造把握できる）
ratio_y = (alpha == 0).mean(axis=1)
ratio_x = (alpha == 0).mean(axis=0)
# ratio が 0.99 以上連続している区間 = 行/列ギャップ
```

**サンプル抽出だけで「OK」と言わない**。全カテゴリから最低1枚ずつ Read して目視:

- セル境界がコンテンツを切っていないか

## グリッド検出 3 階層

`detect_split_axis` の優先順位:

1. **実ギャップ検出**（標準）
   - 背景比率99%以上の連続行/列を「ギャップ」とみなす
   - 12px以内の薄いコンテンツは sliver として merge
   - ギャップ数が `expected + 1` と一致したら採用
2. **等分割フォールバック**
   - ギャップ数が合わなければ画像長さを `expected` 等分
3. **連結成分検出**（`grid=None` 指定時）
   - scipy.ndimage で binary closing → label
   - 不規則レイアウト・要素サイズが異なる時に使う
   - 結果は読み順（上から下→左から右）でソート

## 失敗パターン（過去事例）

| 症状 | 原因 | 対処 |
|---|---|---|
| バッジが横に切れる | コンテンツbboxをセル境界と誤認 | 等分割 or 実ギャップ検出に切り替え |
| 縦方向に隣セルが混入 | 行高さ不均等なのに等分割した | 実ギャップ検出を使う |
| 要素が複数バラける | 連結成分検出の閉じ込めが弱い | `binary_closing(iterations=4)` を増やす |
| chroma-key 残骸（緑/赤被り） | despill 強度不足 | `despill()` の強度パラメータ調整 |

## 絶対にやらないこと

1. **着手時に等分割を即採用**: 行高さが不均等な可能性を最初に潰す
2. **サンプル数枚の確認で完了報告**: 全カテゴリから最低1枚ずつ目視
3. **境界判断を含む自動処理（連結成分など）の結果をユーザー確認なく確定**: 5要素か6要素か等の本質的な数判定はユーザー確認
4. **venv をプロジェクト配下に新設**: `~/.venvs/img-tools/` を使い回す
