# セッション振り返り（Session Retrospective）

## 概要

ユーザーがセッション内での作業・やり取りを振り返りたい時に起動する。気づき抽出 →
反映先候補の提示 → 人間確認 → 実装までを標準化し、振り返りが次回以降の運用に
確実に反映されるよう支援する。

- **誰が**: 横断スキル。メインセッション or 任意のエージェントが参照
- **いつ**: ユーザーが「振り返って」「セッション振り返り」等と明示要求した時のみ
- **何のために**: 一度きりの反省で終わらせず、memory / improvement-log /
  CLAUDE.md / エージェント定義 / 新規スキルへ確実に反映させる

## トリガー（自然文例）

- 「セッション振り返りお願い」「振り返って」「今回のやり取り見直して」
- 「今回の動き、改善点ある？」「ここまでの作業レビュー」

**自動発火はしない**（自動化すると冗長なノイズになる）。

## 標準構造（固定テンプレ・1モード）

**自己採点は付けない**（定量化せず定性的な気づきのみ出す）。

### 1. 良かった点（3-5項目・短く）
- 意図的にうまく動けた点を具体行動と理由セットで
- 「全体的に良かった」のような迎合的な褒めは禁止

### 2. 詰まった瞬間・二度手間（表形式）

| # | 事象 | 原因 | 先回りできたポイント | 本来すべき動き |
|---|---|---|---|---|

- 「詰まった」= 手戻り・失敗・想定外の確認プロンプト発生
- 「二度手間」= 同じ操作・同じ確認の繰り返し（2回以上）
- 原因は表層症状ではなく構造的要因まで掘る

### 3. 自動化・効率化の余地
- 3回以上繰り返した操作 → スキル化・自動化候補
- 型化できる判断プロセス → テンプレ化候補
- permissions 追加で許可プロンプトを減らせるもの
- ルーティング改善で分岐を減らせるもの

### 4. 次回への改善提案
- **アクション可能な粒度**で書く
- 「気をつける」「注意する」は禁止（検証不能）
- 具体例: 「次回 `.zshrc` を編集した時は、現セッションに反映されないことを即座に明示する」

### 5. 反映先候補（SAFE/RISKY 分類 + カテゴリ別承認）

| カテゴリ | 反映先 | 何を書くか |
|---|---|---|
| memory (feedback) | `~/.claude/projects/.../memory/feedback_*.md` | ユーザーの好み・作業スタイル・コミュニケーションパターンの発見 |
| improvement-log | `data/improvement-log.jsonl` | プロセス失敗・検出ルール追加・次回監視したい指標 |
| CLAUDE.md | `CLAUDE.md` | ルーティング表の盲点・MCP運用ルール・人間確認ルールの穴 |
| エージェント定義 | `.claude/agents/**/*.md` | ツール権限不足・参照スキル漏れ・手順明示不足 |
| 新規スキル化候補 | `.claude/skills/` | 3回以上繰り返し発生した手順・型化できる判定プロセス |

#### SAFE 判定（まとめ承認可）
- memory への追記（新規 or 既存 feedback_* への追記）
- improvement-log.jsonl への追記
- CLAUDE.md の文言調整・1行単位の修正
- エージェント定義の文言調整・参照スキル追加・ツール権限の明示追記

#### RISKY 判定（1件ずつ承認）
- CLAUDE.md の**ルーティング表大幅変更・戦略KGI・人間確認ルール**変更
- エージェントの**新規追加・削除・統合**
- **permissions 変更**（新規MCP・Bash除外追加等）
- **新規スキルファイルの確定追加**（CLAUDE.md スキル一覧登録を含む）
- 複数ファイルにまたがる構造変更

## 実行フロー

### Step 1 — 対象スコープの把握
- デフォルト: 直近のセッション全体
- ユーザーがフェーズ指定した場合はそこだけ（例: 「Firecrawl導入の部分だけ」）
- 過去セッション・マルチセッション横断は対象外（これは `monthly-audit.sh` の領域）

### Step 2 — 観点に沿って抽出
- **先回り予測**: 事前に予見できた問題か
- **二度手間**: 同じ確認・操作の繰り返しが発生したか
- **自動化/効率化余地**: スキル化・permissions 追加・ルーティング改善で防げるか

### Step 3 — 対話内で振り返り表示
標準構造テンプレ 1〜5 を対話内に表示。**この段階では何も書き込まない**。

### Step 4 — 反映先候補の提示と承認
5カテゴリ別に反映候補を並べ、SAFE/RISKY を分類して提示:

```
## 反映候補

### SAFE（まとめ承認）
- [memory] feedback_xxx.md に追記: 「〇〇の時は先回りで明示する」
- [エージェント定義] secretary.md のツール権限に 〇〇 追記
→ 全部反映 / 選択反映 / 反映しない?

### RISKY（1件ずつ承認）
1. [新規スキル化] 〇〇スキル追加 — 承認しますか?
2. [permissions] 〇〇 を allow に追加 — 承認しますか?
```

### Step 5 — ファイル化と実装
- ユーザー承認後、反映実装
- 振り返り本体を `outputs/retrospectives/YYYY-MM-DD-HHmm[-topic].md` に保存
  - `HHmm` は24時制で上書き回避
  - `topic` は任意の短い要約（lowercase + hyphen、日本語は避ける）
  - 例: `2026-04-23-1545-plugins-mcp-integration.md`
- ファイル冒頭に日時・対象セッションの要約を入れる

## 既存仕組みとの関係

| 既存 | 関係 |
|---|---|
| `cost-control.md` | コスト観点は参照。振り返り本体の主眼はプロセスの質 |
| `daily-scan.md` / `asana-management.md` / `task-sync.md` | 日次運用系。振り返りとは別文脈 |
| `claude-md-health-check.md` | CLAUDE.md 修正反映時に併用推奨 |
| `agent-onboarding.md` | 新規エージェント化が反映先に出た時に併用必須 |
| `scripts/monthly-audit.sh` | 月次全体監査の補完（競合しない・粒度が違う） |
| `secretary.md` の改善提案審査モード | SAFE/RISKY 分類を同型で採用（一貫性確保） |

## 絶対にやらないこと

1. **人間確認なしに memory / CLAUDE.md / エージェント定義 / permissions を書き換える**
   → 全ての反映は人間承認フローを経る
2. **迎合的な振り返り**（「全体的にうまくいきました」）を出す
   → 具体行動・理由・反省材料を必ずセット
3. **具体性のない改善提案**（「次はもっと注意します」「気をつけます」）を出す
   → アクション可能な粒度必須
4. **振り返りをファイル化せず対話内だけで終わらせる**
   → 必ず `outputs/retrospectives/` に保存
5. **反映候補を出さずに振り返りを終える**
   → 気づき抽出が目的ではなく、次回以降への反映が目的
6. **SAFE/RISKY 判定を省略する**
   → 承認負荷が高まり反映率が下がる
7. **ユーザーに自己採点を求める**
   → 自己採点は仕様上不要、評価させない
