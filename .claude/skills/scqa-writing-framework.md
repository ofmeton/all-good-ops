# SCQAライティングフレームワーク（SCQA Writing Framework）

## 概要
McKinsey 由来の「Situation → Complication → Question → Answer」で文章を構造化するフレームワーク。未整理のアイデアを、論理的で引きのある読み物に変換する。記事・スレッド・提案書・報告書の導入設計に有効。

## 対象
- writer（記事・企画書・報告書 / メイン）
- brand-publisher（ofmeton名義の発信 / SNS・ブログ）
- message-crafter（初回提案・営業文面の導入設計）
- strategic-advisor（報告・論点整理の構造化）
- rapid-hp-operator（BSA提案文のフック設計）
- presentation-reviewer（プレゼン構成のレビュー観点）

---

## 4要素の役割

### S — Situation（状況）
- 読者と共有できる前提・現状を置く
- 「これは自分の話だ」と読者に認識させる
- 関連性のある事実だけを簡潔に。背景説明の長文化は禁止

### C — Complication（問題・転機）
- 状況を崩す変化・矛盾・緊張を導入する
- 読者の中に「このままだとマズい / 何かズレている」を生む
- Complication が弱いと文章全体が退屈になる。ここで温度を作る

### Q — Question（問い）
- Complication から自然に導かれる中心の問いを言語化
- 読者が無意識に抱き始めた問いを、先回りして代弁する
- 「では、どうすれば？」が典型形。問いは1つに絞る

### A — Answer（答え）
- Question への直接的な回答 / 主張 / 解決策
- 明確で、実行可能・検証可能な粒度
- ここが記事・スレッド・提案の本体につながる導線になる

## 典型フォーマット

### ミニマム型（SNS・スレッド冒頭・リード文）
```
[Situation] 〜という状況がある。
[Complication] ところが〜という変化 / 落とし穴がある。
[Question] では、〜はどうすべきか？
[Answer] 結論は〜。本稿ではその理由を3つ示す。
```

### 記事・レポート構成型
```
## リード（S + C + Q を3段落以内）
## 結論（A を先出し）
## 本文
  - 根拠1
  - 根拠2
  - 根拠3
## 実行に移すには（A を行動に落とす）
```

### 提案・営業文面型
```
[S] 貴社の現在の状況（相手が認識している事実）
[C] このままだと起こり得る損失 / 機会損失
[Q] ではどう対処するか？
[A] 私が提供できる解決策（商品ライン・価格・納期）
```

## 使い分けガイド

| 目的 | 重心 | 備考 |
|---|---|---|
| SNS投稿・スレッド冒頭 | C を強く | 3秒で離脱判断される。Complication で止める |
| ブログ記事 | S→C→Q→A をリードで完結 | SEOクエリが Question 相当になるケース多い |
| 営業・提案文面 | A の具体度 | Answer に価格・納期・SLA まで入れ切る |
| 報告書・週報 | S と A の差分 | 先週の S と今週の A のギャップを強調 |
| プレゼン冒頭 | Q の言い切り | タイトルスライド直後に Q を単独で置くと引きが出る |

---

## 執筆手順

1. ゴール定義：この文章で読者にどう動いてほしいか（A を仮置き）
2. Situation 候補を列挙：読者が「知っている」と感じる事実 3〜5個
3. Complication 抽出：S を崩す変化・矛盾・緊張。弱ければ書き直す
4. Question 絞り込み：C から導かれる問いを1つに。複数出たら1つ残して他は本文素材に
5. Answer 明確化：Q に真っ直ぐ答える。修飾語を削る
6. SCQA ブロックを上から通読し、飛躍・繰り返し・トーンずれをチェック
7. 本文は Answer を支える根拠の順で展開

---

## 品質チェックリスト

- [ ] S は読者が「自分の話」と感じる事実か
- [ ] C は本当に緊張を生んでいるか（「〜が大事です」止まりは弱い）
- [ ] Q は1つに絞られているか
- [ ] A は Q に直接答えているか（横滑りしていないか）
- [ ] S→C→Q→A の順序で飛躍がないか
- [ ] 冒頭3段落以内に SCQA が完結しているか（記事・スレッドの場合）
- [ ] 専門用語が相手のリテラシーに合っているか
- [ ] 同じことを別の表現で繰り返していないか

---

## よくある失敗

| 失敗 | 原因 | 対策 |
|---|---|---|
| Situation が長い | 背景説明を盛り込みすぎ | 読者が既に知っている事実は削る |
| Complication が弱い | 「〜が重要です」で止まる | 「このままだと何が失われるか」を具体化 |
| Question が複数 | 論点が収束していない | Q を1つに絞り、残りは本文の小見出しに |
| Answer が抽象的 | 主張が弱い / 根拠を詰めていない | 実行可能な粒度（数値・手順・期限）まで落とす |
| SCQAを全部同じ強さで書く | 全体が平坦に | C と A に重心。S と Q は軽く |

---

## 制約
- S→C→Q→A のいずれかを省略しない（順序の入れ替えは可、省略は不可）
- 同じ要素を文中で繰り返さない
- 簡潔さを優先。装飾語より構造を優先
