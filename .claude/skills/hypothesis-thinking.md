# 仮説思考（Hypothesis Thinking）

## 概要
構造仮説とボトルネック特定を通じて、複雑な問題を整理し、効果的な打ち手を導く思考フレームワーク。

## 利用エージェント
- strategic-advisor（メイン）
- career-strategist（収入戦略）

---

## プロセス

### Step 1: 問題の構造化

問題をMECE（漏れなくダブりなく）に分解する。

**分解の切り口**:
- プロセス分解（時系列で段階を分ける）
- 要素分解（構成要素に分ける）
- 対立軸分解（二項対立で整理する）

**出力フォーマット**:
```
## 問題の構造化

### 問題定義
[一文で表現]

### 構造分解
[分解の切り口]: [選んだ理由]

├── 要素1: [説明]
│   ├── 1-1: ...
│   └── 1-2: ...
├── 要素2: [説明]
│   ├── 2-1: ...
│   └── 2-2: ...
└── 要素3: [説明]
    ├── 3-1: ...
    └── 3-2: ...
```

---

### Step 2: 仮説の構築

構造化した問題の中から、最も重要なボトルネックを仮説として設定する。

**仮説の条件**:
- 検証可能であること
- 具体的であること（「なんとなく」ではない）
- 覆る可能性があること（反証可能性）

**出力フォーマット**:
```
## 仮説

### 主仮説
「[具体的な仮説]」

### 根拠
1. [根拠1]
2. [根拠2]
3. [根拠3]

### 反論（あえて否定する）
- もし仮説が間違っているとしたら: ...
- 見落としている可能性: ...

### 対立仮説
「[別の可能性]」
```

---

### Step 3: ボトルネック特定

構造の中で最もレバレッジの効くポイント（ボトルネック）を特定する。

**判定基準**:
| 基準 | 質問 |
|------|------|
| 制約度 | ここが改善されると全体が動くか？ |
| 改善余地 | 現実的に改善できるか？ |
| 影響範囲 | 改善の効果は広範か？ |
| 緊急性 | 今すぐ着手すべきか？ |

**出力フォーマット**:
```
## ボトルネック

### 特定したボトルネック
[一文で表現]

### 判定理由
- 制約度: ★★★★☆ — [理由]
- 改善余地: ★★★☆☆ — [理由]
- 影響範囲: ★★★★★ — [理由]
- 緊急性: ★★★☆☆ — [理由]

### 推奨アクション
1. [最優先のアクション]
2. [次のアクション]
3. [予備のアクション]
```

---

### Step 4: 検証計画

仮説を検証するための具体的な計画を立てる。

```
## 検証計画

| 仮説 | 検証方法 | データ源 | 期限 | 成功基準 |
|------|---------|---------|------|---------|
| 主仮説 | ... | ... | ... | ... |
| 対立仮説 | ... | ... | ... | ... |
```

---

## 注意事項
- 仮説は「正解」ではなく「検証の出発点」。間違っていても問題ない
- 構造化が目的化しないよう、「で、何をすべきか？」を常に問う
- ユーザーの直感や経験を軽視しない。「なんとなく」の裏にある洞察を引き出す
