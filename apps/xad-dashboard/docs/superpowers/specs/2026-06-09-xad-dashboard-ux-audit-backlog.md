# xad-dashboard UX 監査 ＋ 機能バックログ（Team C / Phase C1）

> 2026-06-09。対象 = `apps/xad-dashboard` の既存4画面（観測 `/`・キュレーション `/curation`・承認 `/approval`・Runs `/runs`）。
> 上位文書: プラン `x-ui-web-ui-ui-flickering-stream.md`・共有契約 `docs/superpowers/specs/2026-06-09-xad-dashboard-shared-design-contract.md`。
> 本書は C2 実装の優先度根拠。契約のトークン意味マッピング（slate基調 / blue primary / emerald success / rose danger / amber warn / `lib/colors.ts`）を破壊しない範囲で UX を底上げする。

---

## 0. このツールは誰のものか（プロダクト前提）

- **唯一の利用者** = 運用者本人（工藤陸）。X発信システムの「人間ゲート」を担う観測・操作コンソール。マルチユーザー／公開 SaaS ではない（内部ツール＝ライト固定方針は `globals.css` コメントに既述）。
- 利用文脈 = デスクトップ主体だが、**承認・キュレーション・今すぐ投稿はモバイルからも触りたい**（移動中に判断ゲートを通す）。観測/Runs はデバッグ用途でデスクトップ中心。
- 設計の北極星 = 「**いま自分は何をすべきか（承認待ち何件？ 詰まってる工程は？）が一目でわかり、最小操作で安全に前へ進める**」。

## 1. ペルソナ × シナリオ（網羅）

単一ペルソナ（運用者本人）の **3 つのモード**で網羅する。

### モード A: 日次オペレーション（最頻・モバイル可）
1. 「今日やることは？」→ どこかに**承認待ち件数 / 予約状況**のサマリが欲しい（現状: ナビに承認リンクすら無い）。
2. `/approval` で本文を推敲→承認/却下。メディア添付の判断。→ **完成度高**。
3. `/curation` で収集素材を選抜→執筆へ送る。AI 型提案。→ **完成度高**。
4. （新）`/schedule`・`/publish`（A/B 実装）へナビで到達。→ **ナビリンクが C の責務**。

### モード B: 観測・デバッグ（デスクトップ中心）
5. `/`（工程図）で**どの工程が赤/黄か**を俯瞰→ノードクリックで定義/実行ログを確認。→ 機能はあるが**色の意味の凡例が無い**・パネルがモバイルで破綻。
6. `/runs` で実行履歴を時系列に追う→ run 詳細でステージ毎の trace を読む。→ **生テーブル・最も未整備**。

### モード C: 初見・再訪オンボーディング
7. 各画面が**何の画面か / 何ができるか**が1行で分かる（承認・キュレーションは説明文あり。観測・Runs は無い）。
8. ナビで**現在地**が分かる（現状: active 状態なし）。

## 2. 現状ギャップ監査（画面別・観点別）

観点 = 情報設計 / 操作導線 / 状態可視性 / 空・エラー・ローディング / アクセシビリティ / レスポンシブ。

### 2-1. ナビ shell（`layout.tsx` / `globals.css`）— 最優先
| 観点 | ギャップ |
|---|---|
| 情報設計 | リンク3個のみ（工程図 / Runs / Curation）。**承認・スケジュール・今すぐ投稿が無い**＝契約 IA 未充足。順序も契約と不一致。 |
| 操作導線 | 現在地（active）表示なし。どの画面にいるか不明。 |
| 状態可視性 | brand/タイトル以外の情報なし。ヘッダーが画面間で不統一（観測/Runs は独自・curation/approval は別スタイル）。 |
| a11y | `<nav>` ランドマーク無し。active を `aria-current` で伝えていない。 |
| レスポンシブ | 6エントリがモバイル幅で折返し崩れの懸念。 |
| トークン | `globals.css` が雛形のまま（背景白固定・セマンティック変数なし）。フォーカスリング/選択色/スクロールバーの底上げ余地。 |

### 2-2. 観測 `/`（HomeClient / Flowchart / NodePanel）
| 観点 | ギャップ |
|---|---|
| 情報設計 | ノード色（緑/黄/赤/スレート/灰）の**凡例が無い**＝色の意味が伝わらない。画面タイトル/説明なし。 |
| 状態可視性 | `latest` 取得失敗時 `{}` で全ノード灰になるが、エラーか未実行か区別不能。 |
| 空・ローディング | Flowchart の lazy loading は「読み込み中…」のみ（gray）。NodePanel 実行タブの空状態はある。 |
| a11y | NodePanel 閉じる `×` ボタンに focus 管理なし。tab が `border-black` で contrast は可だがトークン外。 |
| レスポンシブ | パネル `w-[480px]` 固定→モバイルで本体を圧迫。`flex h-screen` がヘッダー高を考慮せず縦溢れ。 |
| トークン | `text-gray-500` / `bg-gray-50` / `text-red-600` 等、契約の slate/rose 体系外。 |

### 2-3. キュレーション `/curation`
| 観点 | 評価 |
|---|---|
| 総合 | **完成度高**。sticky 操作バー・タブ・フィルタ・空状態・AIダイアログ・トークン適合すべて良好。 |
| 微修正 | MaterialCard が `border-gray-*` / `bg-green-*` 等の素の Tailwind 色を使用（契約 slate/emerald と微差）。tablist の `aria-controls`/パネル id 紐付けなし。ダイアログのフォーカストラップ無し。 |

### 2-4. 承認 `/approval`
| 観点 | 評価 |
|---|---|
| 総合 | **完成度高**。トークン適合・sticky ヘッダー・空状態・編集/保存/承認導線・リスクバッジ良好。 |
| 微修正 | メッセージが画面上部固定で、下方カード操作時に見えない（toast 寄せ or カード近傍 feedback 余地）。承認ボタン disabled 理由は title 提示済。 |

### 2-5. Runs `/runs` ＋ `/runs/[id]`— 最も未整備
| 観点 | ギャップ |
|---|---|
| 情報設計 | 生 `<table>`。タイトル説明なし。status が素テキストで色分けなし。 |
| 操作導線 | run 詳細へのリンクは日時のみ。一覧へ戻る導線なし（詳細ページに back link 無し）。 |
| 状態可視性 | status / outcome の色分け無し。duration の可視化なし。 |
| 空・エラー | **空状態が一切ない**（0件で空テーブル）。 |
| a11y | テーブルヘッダ `<th>` に scope なし。リンク文言が日時のみ。 |
| レスポンシブ | テーブルが狭幅で崩れる。カード化なし。 |
| トークン | `text-gray-500` / 素の border。詳細の timeline は inline style 直書き（colors.ts 不使用）。 |

## 3. 機能バックログ（優先度付き）

P0 = 契約必須/最大価値、P1 = 体験の底上げ、P2 = nice-to-have。C2 は **P0→P1 を実装**、P2 は残課題化。

| # | 優先 | 画面 | 項目 | 受入基準 |
|---|---|---|---|---|
| F1 | **P0** | ナビ | 契約6エントリの統一ナビ shell（sticky/brand/active/`aria-current`/横スクロール） | 全6リンク・現在地強調・全 viewport で崩れない |
| F2 | **P0** | 全体 | `globals.css` トークン整備（セマンティック CSS 変数 + focus-visible/選択色/スクロールバー、ライト固定維持） | 契約の意味マッピング不変・全画面の地色統一 |
| F3 | **P0** | Runs 一覧 | カード/テーブル再設計（status バッジ色・空状態・タイトル説明・レスポンシブ） | 0件で空状態・status が emerald/rose/amber/slate で色分け・モバイルで読める |
| F4 | **P0** | Runs 詳細 | timeline 再設計（colors.ts で色決定・back link・status バッジ・空状態） | 一覧へ戻れる・色が colors.ts 由来・空 trace で空状態 |
| F5 | **P1** | 観測 | 色凡例（legend）+ 画面ヘッダー（タイトル/説明） | 5色の意味が画面上で分かる |
| F6 | **P1** | 観測 | NodePanel をモバイルでドロワー化 + トークン slate 統一 + 閉じる a11y | 狭幅で全幅ドロワー・Esc/フォーカスで閉じれる |
| F7 | **P1** | 観測 | `flex h-screen` をヘッダー高考慮に修正（縦溢れ解消） | 工程図がヘッダー下にぴったり収まる |
| F8 | **P1** | キュレーション | MaterialCard の素 Tailwind 色を契約トークンへ寄せる | 見た目同等・slate/emerald/amber へ統一 |
| F9 | **P1** | MediaModal | フォーカストラップ/初期フォーカス/閉じる a11y（**props/export 不変**） | API 不変・キーボードで開閉操作可 |
| F10 | P2 | 全体 | ダッシュボード的サマリ（承認待ち件数等）をナビ or トップに | （残課題） |
| F11 | P2 | 承認 | メッセージの toast 化（スクロール追従） | （残課題） |
| F12 | P2 | キュレ/承認 | ダイアログ/モーダルのフォーカストラップ徹底 | （F9 で MediaModal は対応） |

## 4. UX 改善方針（実装の指針）

1. **一貫性 > 個別最適**: curation/approval の完成度を基準に、観測/Runs を同じ言語（slate 地・白カード・角丸・トークンバッジ・sticky ヘッダー・空状態パターン）へ引き上げる。
2. **色は意味を持つ**: `lib/colors.ts`（観測ノード）と status バッジ（Runs）を**単一のセマンティック写像**に揃える（ok→emerald / error→rose / warn→amber / skipped→slate / idle→slate-400）。凡例で開示。
3. **モバイルでも判断できる**: 承認・キュレーション・Runs は狭幅で崩れない。固定幅パネルはドロワー化。ナビは横スクロール。
4. **状態を隠さない**: 空 / ローディング / エラーを各画面で必ず描く（Runs の欠落を埋める）。
5. **触れる対象を最小に**: 共有契約のトークン凍結・`MediaModal` API・共通 lib シグネチャを不変に保つ。新規 client nav は `app/components/NavBar.tsx` に隔離（A/B の route と非競合）。
