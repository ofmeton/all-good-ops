## 1. 背景・目的

### 現状の課題

| 項目 | 現状 |
|---|---|
| 出退勤の記録 | スタッフがノートに手書き |
| スプシへの転記 | 月末にオーナー（工藤陸）が手入力 |
| 所要時間 | 7 名 × 25 日 = 約 175 セル / 月の手入力。約 1〜2 時間 |
| エラー要因 | 字が読めない・記入漏れ・転記ミス |
| 給与計算との連結 | 毎月 5 日に GAS が自動実行されるが、データ投入が手動なため締切ギリギリになりやすい |

### ゴール

1. **スタッフ側の手数を増やさず**（ノート手書きと同程度）、出退勤データを電子化する
2. **月末のオーナー手入力をゼロにする**
3. **自己申告ベース**を維持する（リアルタイム打刻ではない）
4. **既存の給与計算 GAS（毎月5日トリガー）を改修なしで活かす**
5. **月運用コストを 100 円未満に抑える**

### Non-goals

- リアルタイム打刻（位置情報・GPS 連動・店舗 Wi-Fi 連動）
- 不正打刻の自動検出
- シフト希望提出フォーム（シフト管理は本 spec の範囲外。同スプシのシフト行は別途運用）
- スタッフ間のコミュニケーション機能（チャット、写真共有）
- 給与計算ロジックの変更（休憩控除ルールの追加は本 spec で扱う）

## 2. ユーザー体験

### 通常ケース（happy path）

```
[スタッフ「大津さん」が退勤時に LINE に送信]
大津 → Bot:  「9-17」

[Bot が即返信（reply、無料）]
Bot → 大津:  📝 2026-05-23（金）の勤怠を以下で記録します。

             出勤: 09:00
             退勤: 17:00
             拘束時間: 8時間
             休憩控除: 45分（労基準拠）
             実労働: 7時間15分

             これでよろしければ「OK」と返信してください。
             違う場合は時間を送り直してください。

[スタッフ確認]
大津 → Bot:  「OK」

[Bot が確定処理]
Bot → 大津:  ✅ 記録しました。お疲れ様でした！

[スプシに書き込み]
202605 シート の 大津彩渚 列 / 5月23日 行 → "09:00-17:00"
```

### 訂正ケース

```
[後日訂正]
大津 → Bot:  「先週月曜の退勤17:30に直して」

[Bot がパース＋確認]
Bot → 大津:  📝 2026-05-19（月）の退勤を以下に変更します。

             変更前: 09:00-17:00
             変更後: 09:00-17:30

             これでよろしければ「OK」と返信してください。

大津 → Bot:  「OK」

Bot → 大津:  ✅ 修正しました。
```

### 1日複数勤務（休憩で抜ける）

```
大津 → Bot:  「9-12 と 14-18」

Bot → 大津:  📝 2026-05-23（金）の勤怠を以下で記録します。

             1コマ目: 09:00-12:00（3時間）
             2コマ目: 14:00-18:00（4時間）
             合計拘束: 7時間（休憩2時間は給与計算対象外）
             休憩控除: 0分
             実労働: 7時間

             OK？
```

### エラーケース

| ケース | 挙動 |
|---|---|
| 未登録 userId からの送信 | **何も返信しない**（友だち追加された一般人/誤フォロー対策） |
| パース不可（例: 「お疲れ様」） | 「すみません、時間を読み取れませんでした。例: 9-17、9:00-17:30」と返信 |
| OK 待ち中の二重送信 | 直前の確認状態を破棄して新しい入力を確認状態に遷移 |
| 出勤 > 退勤（誤入力） | 「出勤が退勤より遅いです。確認してください」と返信 |
| 日付指定が未来 | 「未来の日付は記録できません」と返信 |
| Anthropic API エラー | 「一時的にうまく動いていません。少し時間を置いて送り直してください」と返信、オーナーに通知 |

## 4. データモデル

### 4.1 出退勤シート（既存 `1CUJOC4i_...` / `YYYYMM` シート）

**改変なし**。既存の 1 セル 1 時間レンジ形式（例: `09:00-17:00`）に Bot が書き込むだけ。
1 日 6 行ブロック構成（複数シフト対応）も既存仕様を踏襲。

### 4.2 スタッフマスタシート（新規追加）

シート名: `staff_master`

| 列 | 内容 | 例 |
|---|---|---|
| A | スタッフ名 | 大津彩渚 |
| B | LINE userId | `U1234abcd...` |
| C | 出退勤シートの該当列 | E（大津の列） |
| D | 在職フラグ | TRUE / FALSE |
| E | 1日あたり日次レート制限 | 10（デフォルト） |
| F | 備考 | "2026-05-23 onboard" |

- userId は LINE 公式アカウント Manager で取得（初回登録時に「友だち追加」イベントから自動採取）
- 列マッピング C は手動入力（スプシ管理者 = 工藤陸）

### 4.3 pending state シート（新規追加）

シート名: `pending_state`

| 列 | 内容 | 例 |
|---|---|---|
| A | userId | `U1234abcd...` |
| B | 対象日付 | 2026-05-23 |
| C | 対象シフト枠 | new \| edit |
| D | パース結果 JSON | `{"shifts":[{"start":"09:00","end":"17:00"}]}` |
| E | 確認メッセージ送信時刻 | 2026-05-23 17:05:32 |
| F | 状態 | awaiting_ok \| confirmed \| cancelled \| expired |

- 「OK」待ちのメッセージを一時保存
- TTL: 30 分（30分後に未確認なら expired にして「確認期限切れです」と通知）
- 確定 / 期限切れ後は別シート `pending_state_archive` に移動（後から監査可能に）

## 5. 主要フロー詳細

### 5.1 新規打刻フロー

```
1. LINE → webhook POST
2. event.source.userId を staff_master で照合
   - 未登録 → 何もしない（即 200 返却）
3. 日次レート制限チェック
   - 直近 24 時間ウィンドウの打刻メッセージ数 > limit → 「制限超過」reply
4. メッセージ本文を Claude Haiku に渡してパース
   - prompt: §6 参照
   - 失敗 → 「読み取れませんでした」reply
5. パース結果を pending_state に書き込み（awaiting_ok）
6. 確認メッセージを LINE に reply
7. 同じ userId に既存 awaiting_ok があれば cancelled に更新（最新を優先）
```

### 5.2 OK 確定フロー

```
1. LINE → webhook POST（「OK」「ok」「はい」等）
2. event.source.userId で pending_state を引く
   - awaiting_ok のレコードを取得
3. なければ「確認待ちの記録はありません」reply
4. あれば:
   a. 出退勤シートの該当セルに書き込み（時間レンジ形式）
   b. pending_state を confirmed に更新 + archive 移動
   c. 「✅ 記録しました」reply
```

### 5.3 訂正フロー

```
1. LINE → webhook POST（「先週月曜17:30に直して」）
2. Claude Haiku に「日付指定パース + 時刻パース」を依頼
   - 出力: { target_date, shifts[], operation: "edit" }
3. 既存セルの値を取得（変更前として確認メッセージに含める）
4. pending_state に書き込み（operation: edit）
5. 確認メッセージを reply
6. 「OK」受信で確定（既存セルを上書き）
```

### 5.4 不正/エラーガード

| ガード | 実装 |
|---|---|
| userId allowlist | staff_master 未登録は即無視 |
| 日次レート制限 | スタッフ 1 人 1 日 N 通（デフォルト 10、staff_master で調整可） |
| 未来日付拒否 | パース後に `target_date > today` で reject |
| 過去 N 日制限 | デフォルト 7 日。それ以前は「オーナーに直接申告してください」reply |
| Anthropic 予算上限 | Anthropic コンソールで月 $5 上限を設定（保険） |
| LINE webhook 署名検証 | X-Line-Signature ヘッダで HMAC-SHA256 検証（必須） |

## 6. Claude パース仕様

### 6.1 モデル選定

- **claude-haiku-4-5-20251001**（Claude Haiku 4.5）
- 理由: 1メッセージあたり 100〜300 tokens 程度で十分。応答 1〜2 秒。コスト 0.05 円/回前後。
- 構造化出力（JSON）を Tool Use または response prefix で受け取る。

### 6.2 プロンプト（system）

```
あなたは飲食店スタッフの出退勤メッセージから、構造化データを抽出する専門家です。

入力: スタッフがLINEで送った自由文（日本語）
出力: 以下のJSONスキーマに厳密に従ったJSONのみ

スキーマ:
{
  "intent": "record" | "edit" | "confirm" | "cancel" | "unknown",
  "target_date": "YYYY-MM-DD" | null,       // 省略時は本日
  "shifts": [
    { "start": "HH:MM", "end": "HH:MM" }
  ],
  "raw_user_message": "元の入力をそのまま",
  "confidence": 0.0-1.0,
  "ambiguity_note": "曖昧な点があれば日本語で簡潔に"
}

ルール:
- 「9-17」「9時から5時」「朝9時 夕方5時」は全て 09:00-17:00 と解釈
- 「5時」が午前/午後どちらか不明な場合は飲食店の勤務時間帯（11:00-23:00）を優先
- 「OK」「はい」「お願いします」だけのメッセージは intent: "confirm"
- 「キャンセル」「やめる」は intent: "cancel"
- 時刻を一切抽出できない場合は intent: "unknown" + ambiguity_note に理由
- 日付指定（「昨日」「先週月曜」）は target_date に変換。本日は ${today}

例:
入力: 「9-17」 → {"intent":"record","target_date":null,"shifts":[{"start":"09:00","end":"17:00"}],"confidence":0.95,...}
入力: 「先週月曜の退勤17:30に直して」 → {"intent":"edit","target_date":"2026-05-19","shifts":[{"start":null,"end":"17:30"}],"confidence":0.9,...}
入力: 「9-12と14-18」 → {"intent":"record","shifts":[{"start":"09:00","end":"12:00"},{"start":"14:00","end":"18:00"}]...}
```

### 6.3 出力検証

- JSON.parse 失敗 → intent: "unknown" にフォールバック
- 必須フィールド欠落 → 同上
- shifts[].start > shifts[].end → 「出勤が退勤より遅い」エラー
- target_date が未来 → 拒否
- confidence < 0.5 → 「自信がない」として再確認メッセージ

## 7. 既存システムへの改修

### 7.1 給与計算 GAS

**改修不要**。既存の `payroll_monthly_automation.gs` は出退勤シートを読み取って集計するため、書き込み元が手入力か Bot かを問わない。

### 7.2 休憩控除ルールの追加（推奨：今回同時に入れる）

労基準拠で集計時に自動控除:
- 拘束 6h 以下 → 控除 0 分
- 拘束 6h 超〜8h 以下 → 控除 45 分
- 拘束 8h 超 → 控除 60 分

実装場所: 既存の集計 GAS の「総実動」計算部分。
- Phase 1 で Bot 本体と同時に実装する（給与計算の正確性が落ちないため）。
- 実装計画フェーズ（writing-plans）で既存コードを read し、現状ロジックとの差分を確定する。既に控除処理が入っている場合は確認のみで済む可能性あり。

## 8. セキュリティ・運用ガード

### 8.1 シークレット管理

GAS の Script Properties に格納:
- `LINE_CHANNEL_ACCESS_TOKEN`
- `LINE_CHANNEL_SECRET`
- `ANTHROPIC_API_KEY`
- `OWNER_USER_ID`（工藤陸の userId / 重大エラー通知先）

### 8.2 LINE webhook 署名検証

LINE が送ってきた POST には `X-Line-Signature` ヘッダがある。
GAS で HMAC-SHA256 検証 → 不一致なら 401 相当を返さず、サイレント無視（情報漏洩防止）。

### 8.3 監査ログ

シート `audit_log` を別途用意:

| 列 | 内容 |
|---|---|
| A | timestamp |
| B | userId |
| C | スタッフ名 |
| D | message_in（受信した本文） |
| E | intent / shifts JSON |
| F | message_out（送った返信） |
| G | result（success / error / ignored） |
| H | error_detail |

すべての webhook 受信を 1 行ずつ追記。
3 ヶ月以上前のログは月次クリーンアップでアーカイブシートへ移動。

### 8.4 オーナー通知

以下が発生したら GAS で工藤陸の LINE userId に push notification（無料 200 通枠を使用）:
- Anthropic API が 3 回連続でエラー
- 未登録 userId からの送信が 1 日 5 回以上（スパムフォロー疑い）
- レート制限超過

## 9. テスト戦略

### 9.1 単体テスト（GAS には Jest がないので簡易フレーム）

- `parseTime("9-17")` → `[{start:"09:00",end:"17:00"}]`
- `parseTime("9:00 - 17:30")` → `[{start:"09:00",end:"17:30"}]`
- `applyBreakDeduction({restraint: "8h"})` → 45min
- `applyBreakDeduction({restraint: "8h01m"})` → 60min
- `verifySignature(rawBody, signature, secret)` → true/false
- `checkRateLimit(userId)` → boolean

### 9.2 統合テスト

LINE Bot 用テストツール（または curl で webhook を直接叩く）で:
1. allowlist 外 userId からの送信 → 無視されること
2. 正常な「9-17」→ 確認メッセージ → 「OK」→ スプシ書き込み
3. 訂正フロー
4. 確認待ちのまま 30 分経過 → expired
5. レート制限超過

### 9.3 受け入れテスト（実機）

工藤陸が自分の LINE で 1 日テスト:
- 朝の出勤打刻
- 退勤打刻
- 訂正
- 複数シフト
- 不正入力（「お疲れ様」「9から」など曖昧）

## 10. 移行計画

### 10.1 ロールアウト

ユーザー回答: **ノートと入れ替えに 1 日で切替**

#### Day -7 (準備)
- 新規 LINE 公式アカウント作成（同 Business ID 配下）
- Channel access token / secret 取得
- 友だち追加用 QR コード印刷（店舗掲示用 + スタッフ配布用）
- GAS デプロイ、Webhook URL を Channel に登録

#### Day -3 (リハーサル)
- 工藤陸が 1 人で全パターンを実機テスト
- スタッフマスタに 7 名の userId を仮登録（QR 経由で友だち追加してもらった順に）

#### Day 0 (切替日)
- 全スタッフに口頭/LINE で説明（5 分程度の説明文を用意）
- 当日のシフト終わりから Bot 運用開始
- ノートは持参しない（誤運用防止）

#### Day +7 (チェックポイント)
- 1 週間の audit_log を確認
- パース精度・受け入れ度をスタッフにヒアリング
- 必要なら Phase 2 計画策定

### 10.2 切替スクリプト（説明文テンプレ）

```
お疲れ様です。
本日 5/X から出退勤の記録方法が変わります。

【新しいやり方】
1. QR コードで LINE 公式アカウント「RICE CREAM 勤怠 Bot」を友だち追加
2. 退勤後（または出勤時と退勤時）に時間を Bot に送信
   例: 「9-17」「9:00-17:30」「朝9時から夕方5時まで」
3. Bot が確認メッセージを送るので「OK」と返信
4. これだけで月末の集計まで自動で完了します

ノートは本日以降使いません。
質問があれば工藤までお気軽にどうぞ。
```

## 12. 残課題 / Phase 2 候補

### 12.1 Phase 1 で扱わない（明示的に外す）

- シフト希望提出（Bot からシフト希望を出す機能）
- 出勤忘れリマインド（シフト時刻になっても打刻なしならリマインド push）
- 月次集計レポートのスタッフ送信（給与明細 PDF の自動配信）

### 12.2 Phase 2 で検討

- シフト予定との差分検知 → Slack/メール通知
- 「未来の予定」入力対応（シフト希望としての投稿）
- マルチ言語対応（外国人スタッフ採用時）
- 写真添付（領収書を一緒に送る等の業務連絡用途）

## 13. 決定済み事項サマリ

| 論点 | 決定 |
|---|---|
| ホスティング | GAS（Google Apps Script）単独 |
| LINE アカウント | 新規専用 Channel（同 Business ID 配下） |
| 自然言語パース | Claude Haiku 4.5 |
| 休憩時間 | スタッフは総拘束時間で送信、GAS で労基準拠控除（6h超=45分、8h超=60分） |
| 送信タイミング | 出退勤時の 2 回送信が基本、後日まとめ送信も許容 |
| 訂正フロー | Bot に自然文で訂正依頼 |
| 休日 | 何も送らなくて OK（空欄=休日） |
| 移行 | ノートと入れ替えに 1 日で全員切替 |
| 確認ループ | 必須（誤入力防止のため OK/再入力で 1 ターン挟む） |
| 月コスト目標 | 100 円未満 |

---

## 14. 用語

- **拘束時間**: 出勤から退勤までの総時間（休憩を含む）
- **実労働時間**: 拘束時間から休憩を引いた時間
- **pending state**: スタッフが時間を送信した後、「OK」確定が来るまでの待機状態
- **allowlist**: 受け付ける userId の許可リスト。staff_master シートの在職フラグ TRUE のスタッフのみ
