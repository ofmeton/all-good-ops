# 民泊清掃管理アプリ 設計書

- 作成日: 2026-05-14
- 案件種別: 友人からの業務委託（成果物として納品）
- 名義: 工藤陸
- 納期: 2026年6月中旬テスト運用開始
- スコープ: 要件定義書 1〜9章（core機能フルセット）を一括納品。10章「将来展望」（①会計機能 / ②OTA自動連携）は別フェーズ

## 1. 背景・前提

清掃管理業者（友人）向けの民泊清掃管理アプリ。1業者向けの納品物でマルチテナントは不要。
ランニングコストは納品先（友人）負担の前提。運用・保守体制は未定のため、**アカウント名義に依存しないコード（接続情報はすべて環境変数）**にし、後から「友人へ完全移管」「工藤が保守継続」のどちらにも対応できる構成にする。

ロールは3種類:

| ロール | 定義 | 認証方式 |
|---|---|---|
| 管理者 | 清掃管理業者（運営主体） | メール＋パスワード（Supabase Auth） |
| 物件オーナー | 物件の所有者・依頼主 | 物件ごとの固有トークン付きURL |
| 清掃スタッフ | 清掃を実施するスタッフ | スタッフごとの固有トークン付きURL |

## 2. 技術スタック

| 要素 | 構成 |
|---|---|
| フロント／API | Next.js（App Router）。1リポジトリでフロント＋APIルート |
| DB | Supabase Postgres |
| 認証 | 管理者＝Supabase Auth（email/pw）、オーナー/スタッフ＝トークンURL自前検証 |
| 写真ストレージ | Supabase Storage |
| 通知 | LINE Messaging API ＋ メール（Resend） |
| 定時処理 | Vercel Cron |
| リアルタイム | Supabase Realtime（ステータスのリアルタイム表示） |
| ホスティング | Vercel |

選定理由: 1ヶ月の納期に対し認証・DB・ストレージ・cron・リアルタイムが揃う / 工藤の既存の慣れ（ai-radar・terra-isshiki）が活きる / Supabase・Vercel は「Transfer ownership」で友人移管が容易。

## 3. データモデル

```
admins          管理者（Supabase Auth管理）
  id, email, name, role_level, created_at

owners          物件オーナー（人）
  id, name, line_user_id, email, created_at
                ※1オーナーが複数物件を持てる

properties      物件
  id, owner_id, name, address, access_info_note,
  checklist_template (JSON: [{label, type}]),
  created_at, archived_at

staff           清掃スタッフ
  id, name, line_user_id, email, created_at, archived_at

staff_assignments   スタッフ↔担当物件（N:N）
  staff_id, property_id

access_tokens   トークンURL（紛失・退職時に revoke）
  id, token, type('owner'|'staff'),
  property_id (owner用) / staff_id (staff用),
  revoked_at, created_at

cleaning_requests   清掃依頼
  id, property_id, checkin_date, checkout_date, guest_count,
  option_memo, status, assigned_staff_id,
  assignment_deadline (送信+24h), created_by, created_at, updated_at
  status: unassigned → assigned → in_progress → reported → confirmed

cleaning_reports    完了報告
  id, request_id, staff_id, checklist_result (JSON), submitted_at

report_photos       完了写真
  id, report_id, storage_path, uploaded_at, expires_at (+3ヶ月)

supply_requests     備品補充依頼
  id, property_id, request_id(任意), staff_id, items, created_at

notifications_log   通知送信ログ（冪等性・デバッグ用）
  id, channel('line'|'email'), recipient, kind, payload, status, sent_at
```

設計判断:
- 「次回予約人数」は専用カラムを持たず、同物件の次の `cleaning_requests` の `guest_count` を参照して算出。
- チェックリストは物件ごとカスタムのため `properties.checklist_template` に JSON で保持。提出結果も `cleaning_reports.checklist_result` に JSON。規模が小さくテーブル増殖を避ける。
- オーナーは「人」として独立エンティティ。URLは物件単位（`access_tokens`）なので複数物件保有でも自然に物件数分のトークンになる。

## 4. 認証・認可

**設計判断: 全DBアクセスを Next.js サーバー経由（Supabase service role key）に統一し、認可はアプリ層で一元管理する。**

理由:
- オーナー/スタッフはトークンURLアクセスで Supabase Auth セッションを持たない → RLS（JWTベース）だけでは認可しきれない
- RLS とアプリ層認可の二重メンテはバグの温床。物件<10・スタッフ~10の規模＋1ヶ月納期では割に合わない
- service role key は Server Component / Route Handler に閉じ込め、クライアントには一切露出しない

実装の形:
```
resolveActor(request)  → 誰からのアクセスかを解決
  - 管理者:          Supabase Auth セッションから
  - オーナー/スタッフ: URLの token を access_tokens で検証（revoked_at IS NULL）

各データアクセス関数は actor を受け取りスコープを絞る
  - オーナー:  owner_id が自分の物件のみ・閲覧専用
  - スタッフ:  担当物件 かつ 自分の実施分
  - 管理者:    全件（role_level でさらに分岐）
```

トークン仕様:
- `crypto.randomBytes(32)` の base64url（推測困難）
- 無効化 = `revoked_at` セット / 再発行 = 新トークン行を作り旧トークンを revoke
- URL: `/property/{token}` `/staff/{token}`

## 5. 画面構成・ルーティング

### 管理者（要認証）
- `/admin/login`
- `/admin` — ダッシュボード（カレンダー＋一覧・全物件）
- `/admin/requests` — 依頼一覧（作成/編集/キャンセル/スタッフ割当）
- `/admin/requests/[id]` — 依頼詳細・完了報告の確認（チェックリスト・写真閲覧）・「内容確認済み」操作
- `/admin/properties` — 物件管理（追加/編集/削除・チェックリスト編集・トークン発行/無効化）
- `/admin/staff` — スタッフ管理（追加/編集/削除・トークン発行/無効化）
- `/admin/admins` — 管理者管理（追加/削除・権限レベル設定）
- `/admin/supplies` — 備品補充依頼一覧

### 物件オーナー（トークンURL・閲覧専用）
- `/property/[token]` — 担当物件のステータス・カレンダー・清掃履歴・完了写真/チェックリスト・備品補充依頼履歴

### 清掃スタッフ（トークンURL）
- `/staff/[token]` — 担当物件の依頼一覧・カレンダー
- `/staff/[token]/requests/[id]` — 承認・開始・チェックリスト記入・写真アップロード・備品補充依頼

操作系（承認・割当・ステータス変更・提出など）は Route Handlers 経由。

## 6. ステータスフローと「早い者勝ち」承認の排他制御

```
清掃員未割り当て (unassigned)
  ↓ スタッフ承認（早い者勝ち・有効期限24h）または 管理者が手動割り当て
清掃員割り当て済み (assigned)
  ↓ スタッフがアプリ上で開始
清掃中 (in_progress)
  ↓ チェックリスト・写真を提出して完了報告
完了報告済み (reported) → 管理者へ通知
  ↓ 管理者が内容を確認
内容確認済み (confirmed) → 物件オーナーへ通知
```

早い者勝ち承認は条件付きUPDATEで自然に排他:
```sql
UPDATE cleaning_requests SET status='assigned', assigned_staff_id=:sid
WHERE id=:id AND status='unassigned'
```
影響行数 1 → そのスタッフ確定 / 0 → 「先に取られました」表示。追加のロック機構は不要。

その他のバリデーション:
- 24h有効期限: `assignment_deadline` を作成時にセット。超過＋未割当を cron がスキャンしてアラート。**自動キャンセルはせず管理者が手動対応**（要件通り）。
- 当日割り当て不可（チェックイン/アウトは翌日以降のみ）。
- 翌日割当時、同物件に連続予約があれば警告表示（ブロックはしない）。
- ステータス遷移はサーバー側の状態機械で許可された遷移のみ受け付ける。

## 7. 通知アーキテクチャ

LINE Messaging API ＋ メール（Resend）。

| トリガー | 宛先 | 実装 |
|---|---|---|
| 依頼作成 | 担当スタッフ全員 | 即時（Route Handler内） |
| 完了報告（reported） | 管理者 | 即時 |
| 内容確認済み（confirmed） | 物件オーナー | 即時 |
| 備品補充依頼 | 管理者＋オーナー | 即時 |
| 前日17:00リマインド | 担当スタッフ | Vercel Cron（毎日17:00、翌日の依頼対象） |
| 24h未割当アラート | 管理者＋オーナー | Vercel Cron（1時間ごとにスキャン） |
| 写真の期限切れ削除 | — | Vercel Cron（毎日） |

- 冪等性: `notifications_log` に記録、同一 kind＋対象＋日付の重複送信を防ぐ。
- `line_user_id` 未登録のスタッフ/オーナーはメールのみ送信＋管理画面に警告表示。
- ランニングコスト注記: LINE Messaging API は無料枠あり、規模的に収まる見込みだが要確認。Resend も無料枠内想定。

## 8. 写真ストレージとライフサイクル

- Supabase Storage に保存、`report_photos` にパス記録。
- アップロード時にリサイズ/圧縮（スマホ写真対策）。
- 閲覧は認可済みアクターにのみ短期署名付きURLを発行。
- 自動削除: `expires_at = uploaded_at + 3ヶ月`。毎日の cron で Storage＋テーブル両方から削除。

## 9. エラー処理・テスト方針

### エラー処理
- 無効/revoked トークン → 専用エラーページ（「このURLは無効です。管理者にお問い合わせください」）。
- 通知送信失敗 → `notifications_log` に status=failed 記録＋管理画面表示、処理自体は止めない。
- 排他制御の負け → UIで穏当に通知（「先に取られました」）。
- ステータス競合 → サーバー側の状態機械検証で弾く。

### テスト（優先順）
1. 認可ロジック（`resolveActor`＋スコープ絞り込み）— ユニット最優先
2. 早い者勝ちの並行UPDATE
3. ステータス状態機械の遷移
4. 通知の冪等性（重複送信防止）
5. E2E 主要フロー1本（依頼作成→承認→開始→提出→確認）を Playwright

## 10. 非機能要件

| 項目 | 内容 |
|---|---|
| 対応デバイス | スマホ必須（レスポンシブ・スマホファースト、スタッフ画面は特にモバイル主）。PC・タブレットも利用可 |
| 想定規模 | 物件数10軒未満、スタッフ10名程度 |
| 写真保存 | 3ヶ月保存・期限後自動削除（Supabase Storage） |
| LINE通知 | LINE Messaging API（LINE Notify は廃止済みのため不採用） |
| トークンURL | 推測困難なランダム文字列、管理者が無効化・再発行可能 |

## 11. スコープ外（将来フェーズ）

要件定義書 10章「将来展望」は本納品スコープ外:
- ① 会計機能（スタッフ時給・物件清掃目安時間・交通費/手当・月別報酬算出・月末締め・時給変更時の過去データ保全）
- ② OTA自動連携（Airbnb iCal からの予約自動取得・未割り当て予約一覧・手動入力との重複消し込み）

本フェーズでは将来フェーズ向けの先回り実装はしない（YAGNI）。会計機能が必要になった時点で `cleaning_reports` の `submitted_at` や開始時刻の記録などを追加で設計する。
