# 民泊清掃管理アプリ 設計書

- 作成日: 2026年5月14日
- 制作: 工藤陸
- スコープ: 要件定義書 1〜9章の全機能を一括納品。10章「将来展望」（会計機能 / OTA自動連携）は別フェーズ
- テスト運用開始目標: 2026年6月中旬

---

## 1. 概要

民泊の清掃業務を、清掃管理業者・物件オーナー・清掃スタッフの3者で回すための管理アプリです。
依頼の作成から、スタッフの割り当て、清掃の実施・報告、オーナーへの完了連絡までを1つのアプリで完結させます。

| ロール | 役割 | ログイン方法 |
|---|---|---|
| 管理者 | 清掃管理業者（運営主体） | メールアドレス＋パスワード |
| 物件オーナー | 物件の所有者・依頼主 | 物件ごとの専用URL（ログイン不要・閲覧のみ） |
| 清掃スタッフ | 清掃を実施するスタッフ | スタッフごとの専用URL（ログイン不要） |

- 管理者は複数登録でき、権限レベルを設定できます。
- オーナー・スタッフは個別の専用URLでアクセスします。URLの紛失・退職・契約終了時は管理者がいつでも無効化・再発行できます。

## 2. 機能と画面

### 管理者（要ログイン）
- ダッシュボード：全物件の清掃予約をカレンダーと一覧で確認
- 清掃依頼：作成・編集・キャンセル、スタッフの割り当て・変更
- 完了報告の確認：スタッフが提出したチェックリスト・写真を確認し「内容確認済み」にする
- 物件管理：物件の追加・編集・削除、物件ごとのチェックリストのカスタマイズ、専用URLの発行・無効化・再発行
- スタッフ管理：スタッフの追加・編集・削除（氏名・LINE・メール・担当物件）、専用URLの発行・無効化・再発行
- 管理者管理：管理者の追加・削除・権限レベル設定
- 備品補充依頼の一覧

### 物件オーナー（専用URL・閲覧のみ）
1つのURLにつき1物件の情報のみを閲覧できます。
- 清掃ステータスのリアルタイム表示
- 清掃予約カレンダー
- 直近の清掃履歴一覧
- 清掃完了写真・チェックリスト
- 備品補充依頼の履歴

### 清掃スタッフ（専用URL）
- 担当物件の清掃依頼一覧・カレンダー
- 依頼の承認（早い者勝ち）
- 次回予約人数の確認
- チェックリストの記入・写真のアップロード
- 備品補充依頼の送信

## 3. 清掃依頼のステータスフロー

```
清掃員未割り当て
  ↓ スタッフが承認（早い者勝ち・有効期限24時間）または 管理者が手動で割り当て
清掃員割り当て済み
  ↓ スタッフがアプリ上で清掃を開始
清掃中
  ↓ チェックリスト・写真を提出して完了報告
完了報告済み  → 管理者へ通知
  ↓ 管理者が内容を確認
内容確認済み  → 物件オーナーへ通知
```

**依頼作成のルール**
- 入力項目：物件、チェックイン日、チェックアウト日、宿泊人数、オプションメモ
- 当日割り当ては不可（翌日以降のみ）
- 翌日割り当て時、連続予約が入っている場合は警告を表示

**割り当てのルール**
- 担当スタッフ全員に依頼通知を送信し、最初に承認したスタッフが担当に確定（早い者勝ち）
- 有効期限は依頼送信から24時間
- 24時間以内に未割り当ての場合、管理者・物件オーナーへアラート通知（自動キャンセルはせず、管理者が手動で対応）

**変更・キャンセル**
- 管理者のみ実施可能

## 4. 通知

LINE（LINE Messaging API）とメールの両方で通知します。

| タイミング | 宛先 |
|---|---|
| 依頼作成時 | 担当スタッフ全員 |
| 完了報告時 | 管理者 |
| 内容確認済みになった時 | 物件オーナー |
| 備品補充依頼時 | 管理者・物件オーナー |
| 前日17:00（リマインド） | 担当スタッフ |
| 24時間以内に未割り当て（アラート） | 管理者・物件オーナー |

- LINEユーザーIDが未登録のスタッフ・オーナーには、メールのみ送信し、管理画面に注意表示を出します。
- 鍵・入室方法などの物件詳細情報は、アプリ外（別途連絡）で共有する運用とします。

## 5. データの扱い

- **写真**：清掃完了写真はクラウドストレージに保存し、3ヶ月後に自動削除します。
- **専用URL**：推測困難なランダムな文字列を使用します。管理者がいつでも無効化・再発行できます。
- **アクセス制御**：3つのロールそれぞれに応じて、閲覧・操作できる範囲を適切に制限します。オーナーは担当物件のみ、スタッフは担当物件かつ自分の実施分のみが見えます。

## 6. 技術構成

| 要素 | 内容 |
|---|---|
| アプリケーション | Next.js（Webアプリ。スマホ・PC・タブレットのブラウザで利用） |
| データベース | Supabase（PostgreSQL） |
| 写真ストレージ | Supabase Storage |
| 通知 | LINE Messaging API ＋ メール |
| ホスティング | Vercel |

- 接続情報はすべて設定ファイルで管理し、アカウント名義に依存しない構成にします。納品後の運用体制（オーナー側での運用継続 / 制作者による保守継続）のどちらにも対応できます。
- 品質確保のため、認証・権限まわりを中心に自動テストを整備します。

## 7. 対応デバイス・規模

| 項目 | 内容 |
|---|---|
| 対応デバイス | スマホ必須（レスポンシブ対応）。PC・タブレットも利用可。スタッフ画面は特にスマホでの利用を主に設計 |
| 想定規模 | 物件数10軒未満、スタッフ10名程度 |
| 写真保存 | 3ヶ月保存・期限後自動削除 |

## 8. 今回のスコープと将来フェーズ

### 今回の納品スコープ
要件定義書 1〜9章の全機能。

### 将来フェーズ（今回のスコープ外）
要件定義書 10章「将来展望」：
- **① 会計機能**：スタッフごとの時給設定、物件ごとの清掃目安時間、交通費・手当、月別報酬額の算出（月末締め）、時給変更時の過去データ保全
- **② OTA自動連携**：Airbnb iCal からの予約自動取得、未割り当て予約一覧からの清掃依頼自動入力、手動入力との重複消し込み

将来フェーズは、必要になった段階で改めて設計・お見積りいたします。
