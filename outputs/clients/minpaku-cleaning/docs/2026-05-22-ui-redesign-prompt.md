# 民泊清掃管理アプリ UI Redesign プロンプト

クライアントへの Phase 2 提案用に、Claude.ai の Artifacts で全画面 UI モックを生成するためのプロンプト。

## 使い方

1. https://claude.ai で新規チャット
2. モデルは **Claude Sonnet 4.6 以上**（Opus 4.7 推奨）
3. 下記プロンプトを **そのままコピペ送信**
4. Artifact が生成されたら、各画面ごとに「この画面の余白をもう少し」「ここに〇〇のセクション足して」と追加指示で詰めていく
5. 気に入った最終形を `outputs/clients/minpaku-cleaning/_design_proposal/` 配下に保存（または GitHub Gist）

## クライアント提示時の補足

- 「現状の機能はそのまま」「**見た目だけ全リデザイン**」の提案である旨を明示
- 工数: 全画面リデザインで 15-25h（時給ベース換算で別途見積もり）
- 検収完了後の Phase 2 として提案する

## プロンプト本体（以下を全文コピペ）

````markdown
あなたはモダン SaaS の UI/UX デザイナー兼フロントエンドエンジニア。
民泊清掃管理アプリ（運用中）の UI を「清潔感モダン」方向で全画面リデザインしてほしい。

## プロダクト概要

- 民泊（バケーションレンタル）の清掃管理 SaaS
- 3 ロール:
  - **管理者**（清掃管理業者の運営担当）: PC 操作中心。物件・オーナー・スタッフ・依頼・備品補充の管理
  - **清掃スタッフ**: スマホ操作のみ。トークン URL でログインレスにアクセス
  - **物件オーナー**: PC/スマホ両対応。トークン URL で閲覧専用
- 依頼ステータス: `未割当 → 割当済み → 清掃中 → 報告済み → 確認済み`（+ キャンセル）
- 担当関係: スタッフ ↔ 物件 は N:N（1 スタッフが複数物件 / 1 物件に複数スタッフ）
- 通知: 依頼作成時は LINE 優先、それ以外はメール

## デザイン要件

### トーン

- **清潔感モダン**（Airbnb / Booking.com / Hostfully 系を参考）
- 余白多め、写真の見栄え優先、アクセントカラー 1 色をしっかり使う
- 既存実装は黒ボタン + Tailwind 標準色のミニマル UI だが、**完全に別物としてリデザイン** してよい

### 配色

配色はあなたが決める。下記制約のみ:

- ベースは「白〜オフホワイト系」（疲れず長時間運用できる）
- アクセント 1 色をしっかり使う（民泊・清潔・信頼感を表現）
- ステータスごとに区別できるカラーパレット
  - 未割当 / 割当済み / 清掃中 / 報告済み / 確認済み / キャンセル の 6 状態
- 警告色（連続予約警告など）も別途用意

### ビジュアル要素（必須）

- **lucide-react アイコン**: メニュー、ボタン、ステータス、アクションすべて
- **ダッシュボードのサマリカード**: 「今週の依頼数」「未割当件数」「完了報告待ち」「備品補充未対応」など
- **タイムライン or 週間カレンダービュー**: 物件ごとの清掃予定が時系列で見える（連続予約の視認性 UP）
- **スタッフのアイコン/イニシャル付きアバター**

### ダミーデータの雰囲気

「テストっぽいダミー」で OK:
- 物件 A / 物件 B / 物件 C
- スタッフ 1 / スタッフ 2 / スタッフ 3
- 「合計 12 件」「未割当 3 件」「報告待ち 5 件」など意味のある数値

人物名や住所はあえてリアルにしない（プレゼン時に余計な質問が出ない）。

## 技術スタック

- React (TSX) + Tailwind CSS + lucide-react
- shadcn/ui のコンポーネント風（Card, Badge, Button 等）の inline 実装 OK
- 単一 React コンポーネントで全画面切り替えプレビュー（簡易タブナビゲーション付き）

## 画面一覧（全 15 画面 / 状態）

### 管理者向け（PC 想定 / lg 以上を main、md/sm はサイドバーを drawer 化）

1. **`/admin` ダッシュボード**
   - サイドバー: ダッシュボード / 依頼 / 物件 / オーナー / スタッフ / 備品補充 / 管理者
   - サマリカード 4 枚: 「今週の依頼」「未割当」「報告待ち」「備品補充未対応」
   - 直近の依頼一覧（最大 5 件 / ステータスバッジ付き）
   - 週間カレンダー（物件 × 日付グリッド、各セルにステータス）

2. **`/admin/requests` 依頼一覧**
   - フィルタ: ステータス / 物件 / スタッフ / 日付範囲
   - 表 or カード（テーブルが視認性高い）
   - 各行: 物件名 / チェックイン-アウト / 担当スタッフ / ステータス / 操作

3. **`/admin/requests/[id]` 依頼詳細**
   - 上部: 依頼サマリ（物件・日付・人数・備考）
   - 連続予約警告（黄色アラート: 前後の予約あり）
   - スタッフ割当セクション
   - ステータス操作ボタン（キャンセル / 確認済みにする）
   - 「報告済み」状態時: 完了報告セクション（チェックリスト + 写真ギャラリー + コメント）
   - 通知ログ（LINE/メール 送信履歴）

4. **`/admin/properties` 物件一覧**
   - 各物件: 名前 / 住所 / オーナー / 担当スタッフ数 / 今月の清掃回数 / トークン URL 操作
   - 「新規物件追加」ボタン + フォーム

5. **`/admin/properties/[id]` 物件詳細**
   - 基本情報編集（名前 / 住所 / アクセス情報・備考 / オーナー再割当 / 削除）
   - 担当スタッフ一覧（各スタッフへリンク）
   - オーナー用 URL 発行・無効化
   - 物件専用の清掃履歴（過去 N 件、ステータス付き）

6. **`/admin/staff` スタッフ一覧**
   - 各スタッフ: アバター / 名前 / 担当物件 / 直近の活動 / トークン URL 操作
   - 「新規スタッフ追加」フォーム（担当物件チェックボックス）

7. **`/admin/staff/[id]` スタッフ詳細**
   - 基本情報編集（名前 / メール / LINE ユーザーID / 担当物件チェック / 削除）
   - スタッフ用 URL 発行・無効化
   - スタッフごとの活動履歴（過去 N 件の対応依頼）

8. **`/admin/owners` オーナー一覧**
   - 各オーナー: 名前 / メール / LINE / 所有物件数 / 一覧へのリンク
   - 「新規オーナー追加」フォーム

9. **`/admin/supplies` 備品補充管理**
   - 依頼一覧: 物件 / 依頼者（スタッフ）/ 内容 / 日付 / ステータス（未対応/対応済）
   - 「対応済みにする」ボタン

### スタッフ向け（スマホファースト・縦長レイアウト）

10. **スタッフ依頼一覧**
    - 上部: ヘッダー（スタッフ名 + アバター）
    - 「未割当の依頼」セクション（複数あり）
    - 「自分の割当中の依頼」セクション
    - 各依頼カード: 物件名 + チェックイン-アウト + 「承認する」or「開始する」ボタン

11. **スタッフ依頼詳細（未割当）**
    - 物件情報 + 日付 + 人数
    - アクセス情報・備考（物件特有のメモ）
    - 「この依頼を承認する」ボタン

12. **スタッフ依頼詳細（割当済み・清掃前）**
    - 同上 + 「清掃を開始する」ボタン

13. **スタッフ依頼詳細（清掃中・完了報告フォーム）**
    - チェックリスト（チェックボックス付き）
    - 写真アップロード欄（複数 / プレビュー / 削除可）
    - コメント
    - 「提出する」ボタン
    - 備品補充依頼セクション（入力欄 + 送信）

14. **スタッフ依頼詳細（報告済み・確定後）**
    - 提出済み内容の振り返り表示（チェックリスト結果 / 写真 / コメント）

### オーナー向け（PC/スマホ両対応）

15. **オーナー物件履歴ページ**
    - 物件サマリ（名前 / 住所）
    - 清掃履歴一覧（新しい順 / ステータスバッジ付き / クリックで詳細展開）
    - 完了済み履歴の詳細: チェックリスト結果 + 完了写真ギャラリー + コメント
    - 備品補充依頼の履歴（日付 + 内容）

## 出力形式

1. 単一の React Artifact ファイル
2. 上部に画面切り替え用の簡易タブナビ:
   - `[Admin Dashboard] [Admin Requests] [Admin Request Detail] [Admin Properties] ...` 横スクロール可能
3. 各画面は内部の state で切り替え（実際の routing は不要）
4. 各画面の責務を明確にコメントで残す
5. レスポンシブ対応:
   - Admin: lg 以上想定（PC 最適化）
   - Staff: sm/md 想定（スマホ最適化、縦長フィット）
   - Owner: 両対応

## 注意事項

- 全画面を一気に出すのは難しいので、まず「ダッシュボード + 依頼一覧 + 依頼詳細 + スタッフ依頼一覧 + オーナー履歴」の代表 5 画面から始めてよい
- 全体トーン・配色・コンポーネント方針をその 5 画面で固めて、私（ユーザー）が方向性 OK と返したら残り 10 画面を追加生成
- 段階的に進めるなら「まず代表 5 画面で出して、私のフィードバックを待ってから残りを書く」と最初に宣言してから着手

それでは、まず代表 5 画面のリデザイン案からお願いします。
````

## 進行フロー（推奨）

| ステップ | 内容 | 工藤の作業 |
|---|---|---|
| 1 | 上記プロンプトを Claude.ai (Sonnet 4.6 / Opus 4.7) に投げる | コピペ送信 |
| 2 | 代表 5 画面の Artifact を確認 | 全体トーンと配色を見る |
| 3 | 修正指示（配色変更・余白調整・要素追加など） | 「アクセント色 #XXX に」「サマリカード 4 → 6 に」等 |
| 4 | OK なら「残り 10 画面お願いします」と続行 | - |
| 5 | 全画面揃ったら `outputs/clients/minpaku-cleaning/_design_proposal/` に html/tsx 保存 | - |
| 6 | クライアント提示用にスクショ撮影（playwright で自動化可）| - |
| 7 | Phase 2 提案メール + 見積もり | - |

## 注意

- Claude.ai の Artifact は 1 メッセージで生成できるコード量に上限がある（実用上 1500-2000 行程度）
- 全 15 画面を 1 発で出すのは厳しい → **代表 5 画面 → 残り 10 画面** の 2 ステップ分割が現実的
- shadcn/ui の本物コンポーネントを使うと artifact 内で読み込めないので、inline スタイルで「shadcn 風」に書いてもらう想定
