## B. 仮意思決定中（依頼者の確認をお願いしたい事項）

### B-1. ホームページの役割

- **仮決定**: ハイブリッド型（ブランドサイト + 直接問合せ受付）
- **意味するところ**:
  - 通常の予約は Airbnb 経由がメイン
  - HP は物件の世界観発信・信頼形成・直接問合せ窓口を兼ねる
  - リピーターや法人問合せは HP の問合せフォームから受け付け
- **ご確認**: この方針でよろしいですか？

### B-2. 予約機能のレベル

- **仮決定**: 空室カレンダー表示 + 問合せフォーム（直予約システムは作らない）
- **仕組み**:
  - Airbnb のカレンダー情報を HP に自動取り込み（Airbnb 公式の iCal 機能を使用）
  - HP 上で「埋まっている日／空いている日」が見える
  - 空き日に問合せが来たら大家さんに通知 → 大家さんが Airbnb 側でブロック → メールで承認返信
  - HP 上で決済はしない（Airbnb 経由 or 銀行振込）
- **メリット**: ユーザー体験◎、二重予約リスクは限定的、コストほぼゼロ
- **注意点**: Airbnb のカレンダーと HP の同期に数時間ラグがある（公式仕様）。「目安」と注記する想定
- **ご確認**:
  - この方針でよろしいですか？
  - 将来的に「HP上で決済まで完結する直予約」へ拡張したい構想はありますか？（あれば設計上、拡張余地を残します）

### B-3. 多言語対応

- **仮決定**: 日本語 + 英語の2言語対応
- **理由**:
  - 葉山立地は欧米系インバウンド需要が一定ある
  - 中国語・韓国語まで広げると翻訳運用が重い
- **ご確認**:
  - 英語のみで OK？（中国語・韓国語追加の希望はある？）
  - 英語翻訳はどうしますか？（候補: ① 大家さん側で用意 / ② 工藤側で機械翻訳ベースの下訳 → 大家さんがレビュー / ③ 工藤側で完全に対応）

### B-4. サイト構造

- **仮決定**: シングルページ縦長 LP 型（ホーム1ページに情報を集約）+ 利用規約等の別ページを少数
- **理由**: 民泊サイトの定番パターン。スマホで読みやすく、情報を一気に見せられる
- **ご確認**: マルチページ構成（About / Rooms / Access が別ページ）にしたい希望はありますか？

### B-5. ページ構成（セクション）

シングルページ内に以下のセクションを縦に配置する想定:

1. **Hero**（メインビジュアル + キャッチコピー + 「空室を確認」ボタン）
2. **About**（物件のコンセプト・葉山ストーリー）
3. **Gallery**（プロ撮影写真）
4. **Floor Plan**（間取り・収容人数）
5. **Amenities**（設備一覧）
6. **Location**（地図・アクセス・一色海岸まで徒歩◯分）
7. **Availability**（空室カレンダー + Airbnb予約ボタン + 問合せボタン）
8. **House Rules / FAQ**（チェックイン/アウト・禁止事項・ペット可否等）
9. **Inquiry**（問合せフォーム）
10. **Footer**（住宅宿泊事業届出番号・運営者情報・各種ポリシーリンク）

別ページとして:
- `/terms` 利用規約
- `/privacy` プライバシーポリシー
- `/house-rules` ハウスルール詳細（長文版）

**ご確認**:
- 削りたいセクションはありますか？
- 追加したいセクションはありますか？（例: ブログ・周辺スポット紹介・オーナー紹介・宿泊レビュー）

## C. 依頼者にしか決められない事項（要相談）

以下は制作側で勝手に判断せず、必ず大家さん／依頼者の意向を確認したい事項です。

| # | 項目 | なぜ依頼者判断か |
|---|---|---|
| C-1 | **コンセプト・物件ストーリー文言** | 大家さんの想い・ブランディング方針が出る部分。物件への愛着や経緯を聞いて反映する必要 |
| C-2 | **デザインのトーン**（高級感寄り / カジュアル寄り / ナチュラル寄り） | 物件の客層想定と紐づく。大家さんの好みも大きい |
| C-3 | **ターゲット層**（カップル / ファミリー / 富裕層 / グループ / 撮影利用） | 物件の収益性に直結。コピーや写真選定の方向が変わる |
| C-4 | **Airbnb と HP の動線設計**（直予約割引するか / 同条件か） | Airbnb の規約上、外部誘導には制約あり。リスクと利益を依頼者と共有して決定する必要 |
| C-5 | **法定表示の正確な文言**（住宅宿泊事業届出番号・運営者氏名・連絡先） | 物件の届出書類から正確に転記しないと法令違反になる |
| C-6 | **ドメイン**（terra-isshiki.com / hayama-isshiki.jp 等の選定） | 大家さんの好み・既存ドメイン保有の有無による |
| C-7 | **個人情報の保管方針**（問合せフォーム経由のデータをどこに残すか） | 大家さんの運用と紐づく。プライバシーポリシー文言にも影響 |
| C-8 | **写真の選定優先順位** | プロ撮影写真の中からどれを Hero / Gallery のメインに据えるかは大家さんの感性を反映したい |
| C-9 | **公開予定日・OPEN との連動** | OPEN 日と HP 公開のタイミング設計 |
| C-10 | **制作費・支払条件** | 工藤と大家さんの間で別途協議 |

## D. 工藤側の技術判断で決めて良いと考える事項

以下は技術的妥当性に基づく判断で、依頼者の確認は省略可能と考えます。

| # | 項目 | 工藤判断 | 理由 |
|---|---|---|---|
| D-1 | サイトの基本構成 | 静的サイト（Vercel + Next.js） | B2 レベルなら静的で十分・無料枠で運用可能 |
| D-2 | iCal 連携の実装方法 | 1時間ごとの自動再取得 | Airbnb 公式仕様の範囲で安定動作 |
| D-3 | 問合せフォームの実装 | Resend or Formspree 等の標準サービス | コスト低・運用安定 |
| D-4 | デプロイ環境 | Vercel | 無料枠・SSL自動・高速・運用実績豊富 |
| D-5 | コードベースの構成 | Next.js + TypeScript + Tailwind CSS | 保守性・拡張性が高い |
