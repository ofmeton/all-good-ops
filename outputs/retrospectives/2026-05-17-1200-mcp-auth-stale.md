# セッション振り返り — MCP 認証切れストレス対策

- 日時: 2026-05-17 12:00 頃
- セッション要約: ユーザーから「MCP 認証がいつも切れててストレス」の相談。`claude mcp list` で実態把握 → 原因が プラグイン HTTP MCP の `offline_access` スコープ未付与であることを公式調査 → 棚卸し(A) は後回し、Vercel/Supabase に絞った SessionStart hook(C) を実装

## 1. 良かった点

- **「Google系は実は安定」を即提示できた** — ユーザーの体感（Google系も切れる）を `claude mcp list` で否定し、本当の犯人（プラグイン17個）を最初の1ターンで特定
- **棚卸し提案を「今度」と断られた直後、棚卸し抜きの実用解(hook)に即ピボット** — A案推しを引きずらず、B(実行案)に焦点を絞り直した
- **公式仕様調査をサブエージェント委譲した** — context7 / WebSearch を main で抱えず、claude-code-guide に投げて 400字結論を取り、main の context を節約

## 2. 詰まった瞬間・二度手間

| # | 事象 | 原因 | 先回りできたポイント | 本来すべき動き |
|---|---|---|---|---|
| 1 | 最初の `claude mcp list` が `command not found` | `claude` が `~/.local/bin` 配下で PATH 通ってない実行環境(hook 系と同じ罠) | memory `feedback_nvm_path_for_hooks.md` を冒頭で想起すれば即 `~/.local/bin/claude` で叩けた | Bash 系列で `claude` CLI 呼ぶ時は最初から `~/.local/bin/claude` 固定 |
| 2 | AskUserQuestion を 2回出した(最初の汎用設問 → 棚卸し却下後に hook 対象設問) | 1回目の設問が「どの MCP が切れる？」とユーザー体感を聞きに行ったが、`claude mcp list` で客観確認すれば不要だった | 状況調査(mcp list / cache file 確認)を AskUserQuestion より先に倒すべき | 客観データで答えが出る質問は人間に聞かない。`mcp list` → 結果提示 → 「これを直したいですか？」が正解 |

## 3. 自動化・効率化の余地

- `claude` CLI 呼び出しは常に `~/.local/bin/claude` フルパスでラップ — bash alias か session-env で前置きできれば毎回意識不要
- 「ユーザー体感 vs 実データ」が食い違う系の依頼(今回の「全部切れる」→ 実は一部だけ)は最初に客観確認を入れるパターン化候補。回数増えれば skill 化検討

## 4. 次回への改善提案

- CLI 呼び出し時、`command -v <name>` で見つからなければ `~/.local/bin/<name>` を即試す — `claude` 以外(uvx 等)も同パターン
- 「○○が頻発する」「いつも△△で困る」系の依頼は、AskUserQuestion より先に状態スナップショット(list / status / log)を取る — 体感と実態の乖離をまず潰す
- サブエージェント結論を顧客(ユーザー)提示する前に、結論内の「ユーザーが触れる設定」は1コマンドで再現確認 — 今回なら `claude mcp get plugin:vercel:vercel` で scopes 上書き可否を見るべきだった
