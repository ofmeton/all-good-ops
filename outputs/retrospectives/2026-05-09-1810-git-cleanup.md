## 3. 自動化・効率化の余地

- **初動 sub-repo / build artifact スキャン**: `find . -type d \( -name .git -o -name node_modules -o -name .next \) -not -path "./.git/*"` を Git整理タスクの 1コマンド目に固定
- **gitignore 標準テンプレ**: `node_modules / .next / .turbo / .vercel / dist / build / outputs/**/tmp / outputs/**/_local-check` をテンプレ化して初動で投入
- **Phase 計画時のサイズ概算**: 計画提示前に `git ls-files -o --exclude-standard | xargs du -ch | tail -1` で総量を見て、>100MB or >50ファイルなら分割再検討

## 4. 次回への改善提案

1. Git整理依頼を受けたら、最初の `git status` と並行して `find . -type d \( -name .git -o -name node_modules -o -name .next \) -not -path "./.git/*"` を実行し、初回 AskUserQuestion で sub-repo / build artifact 方針も含めて一度に合意する
2. `.gitignore` を最初に編集する時は build artifacts (node_modules/.next/dist/build/.turbo/.vercel) と scratch dirs (tmp/_local-check/_check) を一括追加 → 後追い拡張を起こさない
3. 新規 .gitignore 行を追加した直後に `git check-ignore -v <代表ファイル>` でサンプル検証 → 漏れの即時発見
4. Phase 計画提示前に `git ls-files -o --exclude-standard <path> | wc -l` と `xargs du -ch | tail -1` を実行し、各 Phase のファイル数・サイズを脚注として明示。閾値超なら分割
