## 1. 観点 A: 論理的一貫性

### A-1: §1.4 比率「翻案 4 : 実体験 4 + 業種別 SOP 2」と §4.7 Hook 比率の整合

- **§1.4**: 翻案 4 / 実体験 4 / 業種別 SOP 2 → "実体験を Phase 1 から本流に"
- **§4.7 Phase 1 Hook 比率**: 数字インパクト 25-30% + 失敗談先行 25-30% + Before-After 15-20% + テスト枠 25-30%

**矛盾の可能性**: §1.4 で「実体験 40%」と言いつつ、§4.7 では「失敗談先行 25-30%」「Before-After 15-20%」の合計でも「実体験」と呼べるのは 50% 程度。但し「数字インパクト 25-30%」は他者事例も含むのが許容され、ここに実体験以外が紛れ込む。

**判定**: 矛盾と言うほどではないが、用語の階層が不明確。**Writer プロンプトで「翻案 vs 実体験 vs 業種別 SOP」と「数字 / 失敗談 / Before-After / その他」を直交軸として明示する** 必要あり。

**修正案 (v10.2 候補)**:
```
2 軸クロス:
                数字  失敗談  Before-After  その他Hook
  翻案 (40%)   ?     ?       ?              ?
  実体験 (40%) ?     ?       ?              ?
  業種別 SOP (20%) ?  ?     ?              ?

Writer プロンプトに毎回「軸 1 と軸 2 を 1 つずつ選んでから生成」ルールを追加
```

### A-2: §4.3.2 fmat 比率変更と §4.8 タイムテーブルの整合

- **§4.3.2 新比率**: 短文 50 / 中文 30 / 長文 15 / スレッド 5
- **§4.8 タイムテーブル**: 1 日 5 投稿 (朝 7:00 / 昼 12:00 / 夕 17:00 / 17:30 / 21:00)

**矛盾**: タイムテーブル上は **すべて短文** を想定している (17:30 / 21:00 は引用 RT で短文中心)。スレッド 5% と長文 15% はタイムテーブルのどこに入る?

**判定**: 矛盾。スレッドと長文の運用枠が §4.8 で明示されていない。

**修正案 (v10.2 候補)**:
```
週次タイムテーブル (月-木):
  通常 5 投稿/日 × 4 日 = 20 投稿/週
  内訳:
    短文 13-14 (65-70%, 朝昼夕全枠)
    中文 5-6 (25-30%, 17:30 引用 RT or 21:00 別角度)
    長文 1 (5%, 木曜 21:00 ロングフォーム)
    スレッド 1 (5%, 水曜 17:00 = note 送客でなくスレッド型告知)

週末:
  金: 4 投稿、短文 + 中文のみ
  土: 3 投稿、短文中心
  日: 4 投稿、Weekly Brief + 来週予告 (長文 1 本含む)
  
週合計: 5 × 4 + 4 + 3 + 4 = 31 投稿
内訳: 短文 22 (71%) + 中文 6 (19%) + 長文 2 (6%) + スレッド 1 (3%)
```

これを §4.8 タイムテーブル詳述として追加する。

### A-3: §4.6 Editor +3 (失敗談 + 実体験ソース ID) と §4.7 Hook テスト枠の整合

- **§4.6 +3**: 失敗談スロット必須、実体験ソース ID 必須
- **§4.7 Phase 1 Hook**: 失敗談先行 25-30% は実体験ベースだが、テスト枠 (問いかけ / 逆張り / 共感 / メタ言及) は実体験ソース ID が不要かもしれない

**判定**: +3 は「失敗談を含む投稿の場合」に限定して読めるが、Editor 実装で「すべての投稿に失敗談スロット必須」と誤読される懸念。

**修正案 (v10.2 候補)**:
```
+3 ルールの厳密化:
- 「失敗談カテゴリと判定された投稿」のみ実体験ソース ID 必須
- 「数字インパクト型」「問いかけ型」など失敗談を含まない投稿は対象外
- Hook 類型分類器 (§4.7) の出力に依存

KPI:
- 月の総投稿のうち fail_rate ≥ 15% (Phase 1 KPI)
- fail_rate 計測対象 = Hook 類型が「失敗談先行型」「経験談導入型」(類型 5)に分類された投稿
```

### A-4: §5.3 異常検知ロールバックと §4.8 transfer learning ingest KPI の整合

- **§5.3**: 平均 PCR -30% 以上で自動ロールバック
- **§4.8 新 KPI**: transfer_learning_ingest_count 週 ≥ 3 件未達は警告

**矛盾の可能性**: transfer learning ingest 不足 → 翻案率低下 → PCR 下落 → 自動ロールバック発火、というカスケード。1 つの根本原因 (ingest 不足) が 2 段階で叩かれる。

**判定**: 二重叩きで問題ないが、**Daily Digest で因果連鎖を 1 行で示す** 必要あり (R-1 と関連)。

**修正案 (v10.2 候補)**:
```
Daily Digest に「PCR 異常検知時の上位因果候補 3 件」を追加:
  例: "PCR -32% → transfer learning ingest が 7 日連続 0 件、翻案率 12% (基準 40%)"
```

## 2. 観点 B: 数値の妥当性

### B-1: 「敬体率 40-55%」の根拠は十分か

- 国内 10 アカ平均 46%、中央値 44%、ばらつき 8% (Shimayus) - 100% (kosuke_agos)
- ofmeton 「翻訳者ポジション」は敬体寄りが妥当だが、40-55% は **中央値 44% に近い** ため "平凡" すぎる

**判定**: 敬体率を ofmeton ターゲット (非エンジニア経営者) に最適化するなら、**55-65%** (敬体寄り) でもいい。理由: 経営者層は「丁寧さ + プロ感」を重視。

**修正案 (v10.2 候補)**:
```
敬体率: 50-60% (前 40-55% → ofmeton ターゲットに最適化)
ただし「失敗談先行型」と「実体験 4 比率」では常体率を 50% まで上げてOK (人間味重視)
```

### B-2: 「投稿頻度 5 件/日」は ADHD/ASD 特性で持続可能か

- 国内 10 アカ中央値 4.7 件/日 (整合)
- ただし R-8 後悔予測 (LINE 480 通/月) の対応として週 1 回まとめインタビューモードを追加したが、**X 投稿そのものの自動生成 + 人間承認の負担** は議論されていない

**判定**: 「自動生成 → 人間承認 → 投稿」のフロー全体で、人間承認の負担を年間 1,500 回 = 月 125 回 = 1 日 4-5 件 こなせるか。

**修正案 (v10.2 候補)**:
```
人間承認の負担軽減策 (Phase 1 のみ):
- Phase 1 Week 1-2: 全投稿人間承認 (5 × 30 = 150 回/月)
- Phase 1 Week 3-4: PCR ≥ 0.3% 期間中の投稿は「自動投稿 + 事後承認 (LINE で reject 可能)」に変更 → 月 75 回承認
- Phase 1 月 2 以降: 「Hook 強度 ≥ 0.5 & cos 類似度 < 0.5 & 6+4 ルール全 pass」を満たす投稿は自動投稿、それ以外は人間承認
- Phase 2 以降: 完全自動 + 異常時のみ人間介入
```

### B-3: 「non_engineer_rate ≥ 30%」を Phase 1 KPI にする閾値の妥当性

- 国内 10 アカ平均 4% (milbon_ 18%、ClaudeCode_love 11%、残り 0-4%)
- 30% は milbon_ よりも +12 ポイント高い、達成可能性は?

**判定**: 達成可能性は高い (Writer プロンプトで強制可能)。**ただし、過剰に「非エンジニア向け」を強調すると、エンジニア層の信頼を失う可能性**。バランス点は 20-25% かもしれない。

**修正案 (v10.2 候補)**:
```
non_engineer_rate Phase 別 KPI:
  Phase 1 (Week 1-4): ≥ 20% (慣らし期間)
  Phase 1 (Month 2-3): ≥ 30%
  Phase 2 以降: 30-40% で推移、自アカ実績で最適化
```

### B-4: 「fail_rate ≥ 15%」と R-2 対応 (実体験ソース ID 必須) のコンビ

- 国内 10 アカ平均 3.3%、ofmeton 目標 15% は **+12 ポイント**
- 月 150 投稿のうち 22-23 投稿が失敗談先行型
- 1 ヶ月で 22 件の "実体験から抽出された失敗談" を Claude Code 履歴 / Git commit / 案件メモ / 音声メモから引っ張ってくる必要

**判定**: ofmeton の活動量 (terra-isshiki / minpaku-cleaning / RICE CREAM / 家庭教師 / portfolio) からして週 5-6 件の失敗談ソース ID は確保可能。**ただし「同じ失敗談を別 angle で 2-3 回語る」許容を Editor に明示しないと、ネタ枯れ**。

**修正案 (v10.2 候補)**:
```
Editor +3 ルールの補足:
- 同じ実体験ソース ID を別 angle で再使用可能 (3 ヶ月以内 max 3 回)
- ただし angle = (時系列の異なる側面 / 違うステークホルダー視点 / 違う Hook 類型) のいずれかが必要
- 直近 2 週で同じ angle で語った場合は §4.6 ルール 5 (類似投稿) で reject
```

### B-5: Style Guide v1 §6.5 ハッシュタグ「≤0.05」の根拠

- 国内 10 アカ hashtag_rate 平均 1.3%、最大 6.9% (minorun365)
- ofmeton 5% は中央値より高めだが妥当

**判定**: 0.05 (= 5%) は妥当。**ただし note 送客時のみハッシュタグ 1-2 個許可** (#AI業務自動化 #Claude #非エンジニア向け) を明示しないと、Writer が 0% に振り切る可能性。

**修正案 (v10.2 候補)**:
```
hashtag_rate: ≤ 0.05 (X 全体)
ただし note 送客投稿 (集客導線 B) のみ #AI業務自動化 #Claude #非エンジニア向け の 3 個まで許可
```

## 3. 観点 C: 実装可能性

### C-1: Editor +3 (実体験ソース ID) の判定方法

- 「実体験ソース ID」を Editor がどう取得 & 検証する?
- Claude Code 履歴 = ローカルファイル / Git commit = SHA / 案件メモ = wiki/projects/ / 音声メモ = ?

**判定**: 実装上、Editor は **Writer 出力に同梱された "実体験ソース ID 配列" を読み取り、Supabase テーブル `materials_store` で存在確認** する設計が現実的。

**修正案 (v10.2 候補)**:
```
実体験ソース ID の形式:
  - Claude Code session ID: "cc_sess_<YYYYMMDD>_<hash>"
  - Git commit SHA: "git_<8 桁 SHA>"
  - 案件メモ slug: "memo_<YYYYMMDD>_<slug>"
  - 音声メモ ID: "voice_<YYYYMMDD>_<hash>"

Supabase テーブル `materials_store` に上記 ID を主キーとして格納
Editor は ID を SELECT し、見つからなければ reject
```

### C-2: 月別業種フォーカス (§1.2) の運用ロード

- 7 月 = 税理士 + 経理 AI、8 月 = 社労士 + 給与計算 AI、...
- 月の素材在庫を業種別に偏らせる必要 = Interviewer 質問パターンも業種別に最適化が必要

**判定**: §4.1 Interviewer の質問パターンライブラリ (8 種) は業種非依存。月別業種フォーカスに合わせて **質問のサブセット選択 + 業種別キーワード注入** が必要。

**修正案 (v10.2 候補)**:
```
§4.1 Interviewer に月別業種フォーカス対応を追加:
  - InterviewState に "monthly_industry_focus" フィールド
  - 値: ["税理士", "経理"] (2026-07)
  - 質問パターン選択時に業種別キーワードを注入
    例: friction_zoom 質問 → "税理士業務で一番時間吸われた瞬間は?"
  
Optimizer の月次ジョブで monthly_industry_focus を自動切替 (§1.2 ローテーション通り)
```

### C-3: Style Guide v1 → Writer プロンプトの固定方法

- v1 の YAML 設定 (敬体率 0.40-0.55、Hook 比率、文字数等) を Writer プロンプトに反映する仕組み
- Phase 1 中に v1.x ('1.1', '1.2' とか?) 微調整するなら version 管理が必要

**判定**: Writer プロンプトは「base prompt + Style Guide ref」の 2 階建てが望ましい。Style Guide 自体は Supabase テーブル `style_guide` に YAML 化 / SHA 管理。

**修正案 (v10.2 候補)**:
```
Supabase テーブル `style_guide` :
  version: '1.0', '1.1', '1.2', ...
  yaml_blob: text
  effective_from: timestamp
  approved_at: timestamp (Style Guide v2 → v3 の人間承認時)
  retired_at: timestamp

Writer は最新 active version を取得 → base prompt に diff を `<style_guide>` セクションで追加
```

### C-4: 集客導線 Phase 1 単純化 (§4.8 R-5 対応) の Phase 2 移行判定

- Phase 1 = A 単独 + 夕方 B 1 投稿
- Phase 2 = 「PCR 3 週連続 + 平均 PCR ≥ 0.4% → B/C の A/B テスト」

**判定**: PCR ≥ 0.4% は国内 10 アカ engagement_top25_avg 平均 0.95% と比べて低めの閾値。達成可能性は十分。ただし **「3 週連続 0.4%」と「平均 PCR」のどっちが優先か** が曖昧。

**修正案 (v10.2 候補)**:
```
集客導線 Phase 1 → Phase 2 移行判定 (AND 条件):
  - 直近 3 週連続で週次 PCR ≥ 0.3% (週次)
  - 直近 30 日の平均 PCR ≥ 0.4% (月次)
  - 上記 2 条件を AND で満たした時点で Phase 2 集客導線 B/C 導入承認伺い
```

### C-5: transfer learning ingest 週 3 件 KPI の運用

- raw/publishing/inspirations/ への ingest は **誰が** やる?
- 自動: twitterapi.io で海外 17 アカ + 国内業種別 7 アカを cron で取得 → wiki ingest
- 手動: ユーザーが LINE / RICE CREAM 業務で見つけた事例

**判定**: §3.1 素材レイヤー (twitterapi.io) で「海外バズ + 公式アカウント追跡」をカバーしているので、cron 化は可能。ただし **wiki/publishing/inspirations/ への ingest プロセス** が v10.1 で明示されていない。

**修正案 (v10.2 候補)**:
```
§3.1 素材レイヤーに「inspirations ingest プロセス」追加:
  weekly cron:
    1. twitterapi.io advanced_search で海外 17 アカ + 国内業種別 7 アカ × 過去 7 日の上位投稿取得
    2. 既存 wiki/publishing/buzz-patterns.md と差分計算
    3. 新規発見を raw/publishing/inspirations/<media>-<date>-<slug>.md として保存
    4. brand-publisher agent が ingest 候補をユーザーに LINE で提示
    5. ユーザー Y/N で wiki/publishing/ に統合
  
  毎週月曜 09:00 JST に Optimizer が ingest count を集計、Weekly Brief に含める
```

## 4. 観点 D: ターゲット解像度

### D-1: ofmeton の「翻訳者」ポジションは "非エンジニア経営者" の何の課題を解決しているか

- v10 §1.2 で「業務仕組み化の翻訳者」
- v10.1 §1.2 で「業種横断翻訳者 (税理士 + 製造 + ...)」

**懸念**: ターゲットの "課題" が言語化されていない。例:
- "AI を入れたいが何ができるか分からない"
- "自社業務に AI が組み込めるか判断できない"
- "AI 導入の見積りが妥当か分からない"
- "AI を入れて失敗するのが怖い"

これらのうち、ofmeton はどれを優先的に解決する?

**修正案 (v10.2 候補)**:
```
§1.2 戦略目的に "ターゲットの 4 大課題" を追加:
  C1: "AI で何ができるか分からない" → 業種別事例で証拠提示 (Tier 1 T1-1)
  C2: "自社業務に AI が組み込めるか判断できない" → 委託フローと要件定義 (T1-2)
  C3: "AI 導入の見積りが妥当か分からない" → 階段単価 + ROI 開示 (T1-3)
  C4: "AI を入れて失敗するのが怖い" → 失敗談先行 (T1-4)

ofmeton の主軸 = C1 + C4 (Phase 1)、C2 + C3 (Phase 2-3)
```

### D-2: §4.3.5 1 トピックの 3 媒体展開フローは "業務時間" 軸でリアルか

- 1 core idea × 3 媒体 = note 1 + X 5 + IG カルーセル 1 + ストーリーズ 1 = 約 8 投稿
- 月 30 core ideas × 8 投稿 = **240 投稿** (v10 §4.3.5 既出)
- 1 ヶ月で 240 投稿の各種素材生成 + 編集 + 投稿 = 何時間?

**判定**: Writer + Editor + 人間承認の合計時間が見積もられていない。

**修正案 (v10.2 候補)**:
```
§4.3.5 末尾に「運用負担見積り」を追加:
  Writer 自動生成: 30 core ideas × 8 投稿 × ~30秒 (MA Sonnet) = ~2 時間 (MA 自動)
  Editor 自動判定: 240 投稿 × ~10秒 = ~40 分 (MA 自動)
  人間承認 (Phase 1): 240 × ~2 分 = **8 時間/月** ← Phase 1 で要管理
  人間承認 (Phase 2 以降): 自動投稿 + 異常時のみ介入 → 1-2 時間/月
  
  Phase 1 着手前に "1 日 15 分 × 30 日 = 7.5 時間/月" 予算を確保することを §8 Phase 計画に追加
```

### D-3: Style Guide v1 §6.4 ofmeton 階段単価が ofmeton 自身に納得感あるか

- L3 個別相談 5,000-10,000 円 = 時給ベースだと 1 時間
- ofmeton の現行家庭教師時給 2,000 円 (R-A 時 2,500 円) と比較すると 2.5-5 倍
- 業務委託として高すぎ or 低すぎないか

**判定**: 5,000-10,000 円は AI 自動化代行の業界相場 (15,000-30,000 円) と比較して **下限ギリギリ**。L4 小型代行 50,000-150,000 円は妥当だが、L3 個別相談はもう少し高めが妥当。

**修正案 (v10.2 候補)**:
```
階段単価微調整:
  L3 個別相談: 8,000-15,000 円 (前 5,000-10,000) — 1 時間 + 後続事例フォロー含む
  L4 小型代行: 80,000-200,000 円 (前 50,000-150,000) — 月 1-3 件
  L5 大型代行: 400,000+ 円 (前 300,000+) — 月 1 件
  
ただし Phase 0 では「枠だけ決め、Phase 1 で個別相談 3-5 件こなしてから確定」(R-6 で既出)
```

## 5. 観点 E: リスクヘッジ漏れ (R-1〜R-10 以外)

### R-11: 「Phase 1 の人間承認 8 時間/月」が ADHD/ASD で続かない

- D-2 で出した推定。実体験 1 日 15 分 × 30 日は ADHD/ASD で「毎日」継続できるか
- 週末まとめ承認モード (R-8 と類似) も必要かも

**修正案 (v10.2 候補)**: §8 Phase 1 に「**承認モード切替**」を追加:
```
Phase 1 承認モード:
  デフォルト: 毎日 15-20 分 × 30 日
  代替: 週 2-3 回 60-90 分 (火 / 金 / 日 等のまとめ承認)
  キルスイッチ: !approve_pause で 7 日間の事前承認 (PCR が下落しない範囲)
```

### R-12: 「業種特化勢 (畠山謙人 / shigyou_ai_com)」との競合が予想以上に厳しい

- v10.1 §1.2 で「業種横断 + 月別フォーカス」で勝つ前提
- 実際は業種特化勢の方が深さで圧倒、ofmeton は浅く広く = 結果 "中途半端"
- E-41 で問うているがリスクヘッジが弱い

**修正案 (v10.2 候補)**:
```
業種特化勢への対応:
  Phase 1 (慣らし): 業種横断で広く認知
  Phase 2 移行判定 (PCR ≥ 0.5% × 3 週):
    Option A: 業種横断を継続 (フォロワー継続増加なら)
    Option B: 業種特化 (1 つの業種に絞る) — 例: "税理士特化" → 畠山謙人と直接対決
    Option C: 横断だが "失敗談アグリゲーター" として独占 (業種ではなくテーマで差別化)
  
  Phase 1 終了時に Optimizer + 人間判断で A/B/C を選択
```

### R-13: note 5 → 3 構成パターンに絞った時、競合 (zeirishi_ai_lab / lush_canna4905 / sevenrich 等) との差別化が薄れる

- 競合は「ツール比較」「段階型」も書いている
- ofmeton が Phase 1 で 3 系統 (まとめ / 専門職×AI / シリーズ) に絞ると、「ツール比較」「段階型」を欲しがる読者が競合に流れる

**修正案 (v10.2 候補)**:
```
§4.3.4.1 補足:
  Phase 1 3 系統に絞る理由 = 書く負担削減 (R-4)
  ただし、Phase 1 中盤 (月 2) で「読者からツール比較を求める声」が PCR / コメントで観察された場合、Phase 2 を待たずに ツール比較 1 本追加可能
  Optimizer Weekly Brief に「未採用構成パターンへの読者反応」項目を追加
```

### R-14: Hook 13 類型のうち未使用 6-7 類型 (警告 / 比較 / 自己卑下 / 経験談導入 / 共感基本 / メタ基本 / みんなXと言うが実はY / 結論先出し) の検証コスト

- v10.1 §4.7 で「Phase 1 主軸 3 + テスト枠 4」に変更
- 残り 6-7 類型 (警告 / 比較 / 自己卑下 / 経験談導入) は Phase 1 で検証されない
- Phase 2 で検証する時、Phase 1 のデータ蓄積を base にできない (= ゼロからスタート)

**修正案 (v10.2 候補)**:
```
§4.7 Phase 計画別 Hook 検証ロードマップ:
  Phase 1: 主軸 3 + テスト枠 4 = 7 類型運用、6 類型は休眠
  Phase 2: 主軸 3 + テスト枠 4 + 休眠から 2-3 類型復活運用 = 9-10 類型
  Phase 3: 13 類型全運用 + 新類型認定 (E-39 で問うた "十分な検証データ" 観点)
```

### R-15: 海外 transfer learning ingest が "そのまま翻案" になり著作権侵害 (R-7 と類似だが異なる)

- §10.2 翻案ルールで「cos 類似度 0.85 以上は翻訳扱い」と定義
- だが、海外 X 投稿は短文 (~280 字) で、cos 類似度を取るにはサンプル小さすぎ
- 「翻案 ≠ 翻訳」を Writer がどう判定するか不明

**修正案 (v10.2 候補)**:
```
§10.2 翻案ルール補足:
  X 短文投稿 (< 200 字) の翻案:
    - 文体・構造を変える (例: 列挙型 → ストーリー型)
    - 固有名詞 (会社名・サービス名) を変える
    - 数字を ofmeton の実例数字に差し替える
    - 上記 3 つすべてを満たさない場合は "翻訳" 扱い
  
  cos 類似度ベースでなく、上記 3 条件の規則ベース判定を Writer プロンプトに追加
```

## 6. 観点 F: 法務 / 規約

### F-1: M-13「数字付き ROI 開示の責任範囲」は §10.1 ステマ規制と整合か

- §10.1: アフィリエイト / 自社販売リンクを含む投稿は明示表記必須
- M-13: 数字付き ROI 開示は自身の実測値のみ、他者事例は出典明示

**統合**: 両者は別軸の規約だが、Editor +2 (ステマ) と Editor +3 (失敗談 + 実体験ソース ID) を統合した時、「他者事例の数字を引用するアフィリエイト誘導 X 投稿」のような複合ケースをどう扱う?

**修正案 (v10.2 候補)**:
```
§10.1 + §10.2 統合ルール (Editor 自動判定):
  IF "数字" exist AND "アフィリエイトリンク" exist:
    REQUIRE:
      - 数字の出典 (実測 or 他者事例 + URL)
      - ステマ明示 ("#PR" / "[PR]" / "[note 販売中]")
    ELSE reject
```

### F-2: 月別業種フォーカス (§1.2) で「税理士」「社労士」「行政書士」「司法書士」「弁護士」を語る時の **業法** 懸念

- 「業務独占資格」を持たない ofmeton が「税理士業務」「社労士業務」を語ると違法行為の勧誘とみなされる懸念
- 「税務相談を AI で」は税理士法違反、「労務相談を AI で」は社労士法違反

**判定**: **重大な未対応リスク**。v10 / v10.1 で全く触れられていない。

**修正案 (v10.2 候補)**:
```
§10 新規 §10.7 業法ガード (重要):
  ofmeton が「税理士業務」「社労士業務」「行政書士業務」「司法書士業務」「弁護士業務」を語る時:
  
  禁止:
    - 個別の税務 / 労務 / 法務相談に AI で回答する内容
    - 業務独占資格を必要とする業務に AI を「代替」する内容
  
  許可:
    - 業務効率化ツール (見積書生成 / OCR / データ整理等) の解説
    - 各士業者本人が自分の業務に AI を使う事例の紹介
    - "業務独占の範囲外" の業務効率化 (経理 / 給与計算等は税理士・社労士業務範囲だが、データ集計等は範囲外)
  
  Editor +5 ルール: "業務独占資格ワード" (税務相談 / 労務相談 / 法務相談 / 訴訟 / 登記) を含む投稿は LLM judge + 規則ベースで reject、ofmeton が "本人代わりに語る" 構造になっていないかチェック
```

これは v10.2 で必須追加。
