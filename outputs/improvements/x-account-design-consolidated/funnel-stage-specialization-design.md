# ファネル段階別 専門化 設計判断 (x-account-design)

> 作成: 2026-06-05 / きっかけ: Kくん @kkk_cun「Claude Codeで5人の月100万円稼ぐnote販売会社」ツイートとの比較
> 区分: 設計判断 (Design Decision Record) / ステータス: 採用

## 0. 一行結論

真似すべきは「フェーズ別チーム」という**箱**ではなく、**フェーズ別の目的関数（KPI + 禁止事項 + 文体）**。
→ **生成フロントを funnel-stage で着せ替え、ガバナンス/インフラのバックは共有のまま。**

---

## 1. きっかけと論点

Kくんの組織は動線の各段階に専任エージェントを置く（縦割り）:

- Xバズポスト作成部隊（流入）
- X交流部隊（流入・アルゴ対策）
- 無料note作成部隊（教育・信頼形成）
- 有料note作成部隊（収益化）

一方こちらは生成が **Writer 1 つ**に集約されている。「段階ごとに目的（流入 / 信頼形成 / 収益）が変わるのに専門性を出させているのは良いのでは」という問題提起。

---

## 2. Kくんの分解の"本当に効いている部分"

専門性の正体は組織図ではなく、**プロンプトの目的が 1 個に絞れていること**。LLM は 1 プロンプトに目標を詰め込むほど出力が薄まる。

| 段階 | 唯一の目標 | 文体 | 禁止事項の核 |
|---|---|---|---|
| 流入 | 読まれる / フォローされる | 140字・フック・リズム | 売り込み |
| 信頼形成 | 教える / 信用される | 1,500字・体系 | 断言・抽象締め |
| 収益 | 買わせる | 販売コピー・価格 | 景表法・再現性断言 |

3 つは KPI も禁止事項も文体も本当に別物。だから割ると各々が濃くなる。**ここは本物の利点。**

現状のこちらの弱点 = 単一 Writer が「良い文章を書く」という曖昧目標で全段階を兼任 → **どの KPI に最適化しているか不明瞭。**

---

## 3. 分解の"軸"の違い（早とちり防止）

| | 分解軸 | 例 |
|---|---|---|
| Kくん | 縦割り（ファネル段階） | 流入 / 信頼 / 収益が各々生成を完結 |
| こちら | 横割り（機能） | Writer 書く → Editor 検閲 → Hook 分析 → Publisher 投稿 → Optimizer 学習 |

こちらは既に機能で割っており（しかも Editor/Optimizer/Publisher 共有で Kくんより上等）。

⚠️ **危険な早とちり**: 「フェーズ別チームを作る = Editor も Publisher も Optimizer も 3 つに複製する」。
段階が違っても検閲・投稿・学習は同じ機構でよい。複製すると drift とメンテ地獄。**複製は無駄。**

---

## 4. 採用方針: ハイブリッド

**生成フロントだけ funnel-stage で profile 分け、ガバナンス/インフラのバックは共有。**

```
[流入Writer] ┐
[信頼Writer] ┼→ 共有Editor → 共有Publisher → 共有Optimizer
[収益Writer] ┘                                  ↑ posterior は段階別
```

- Writer を「1 ロール・3 モード」化（objective + KPI + few-shot + 禁止事項を差し替え）。別エージェント定義でも 1 Writer の profile 切替でも実装は任意。**本質は段階ごとに目的関数と KPI を注入すること。**
- **Optimizer 格上げ**: Thompson Sampling 8 パラを**段階別 posterior** に。流入で効くフックと収益で効くフックは別物。今は混ざっている。"チーム分割"より遥かに価値が高い隠れた upgrade。

---

## 5. 現状の de-facto 専門化と、本当に欠けている 1 本

- こちらは既に**チャネルで割れている**（Writer X / Writer note / Writer IG）。X=流入、note=信頼+収益。→「流入 vs それ以外」は実質分離済み。
- **本当に欠けている線は 1 本だけ**: note 内の「無料note（教育=信頼）」と「有料note（説得=収益）」が同一 Writer だと教育文体と販売文体が混ざる。**真似するならここ。X 側を 3 分割するのは過剰。**

---

## 6. 落とし穴（"いいかも"を冷ます）

1. **Phase1 で稼働中なのは X だけ**。信頼/収益チームを今作っても対象がない → launch を遅らせる方向に効く（完璧主義の癖）。
2. 分割するほど style_feedback / Optimizer 配線が agent 数だけ増える＝**3 倍のメンテ。**
3. 「チームがある」満足感と「KPI が上がる」は別。Kくん本人も結局 1 人で全目視＝箱を増やしても人間の確認負荷は減っていない。

---

## 7. 採用アクション（優先順）

1. **今すぐ・ゼロコスト**: 現 Writer プロンプトに `funnel_stage`（流入/信頼/収益）と**その段階の KPI・禁止事項**を引数注入。エージェントを増やさずモード化。← 利点の 8 割を回収
2. **note 稼働時（Phase1 中盤）**: 無料note-Writer / 有料note-Writer を分離。教育↔販売の文体衝突を解消
3. **Optimizer を段階別 posterior 化**: 本命の差別化
4. **Editor / Publisher / Hook / Visualizer は複製しない**（共有維持）

---

## 8. Kくんの組織から別途盗む点（本論の外・参考）

- **X 交流レイヤー**（いいね/リプ/引用でアルゴを動かす）がこちらの設計に欠けている。soft launch 直後の初期到達を作る要 → Phase1 で交流支援を検討。
- **「動線先・素材後・まず 1 部隊だけ動かす」立ち上げ哲学** = 完璧配線してから launch する癖への解毒剤。

---

## 9. 一言サマリ

「フェーズごとに専門性」の直感は正しい。だが答えは**スタックを縦に増やすこと**ではなく、**1 人の Writer に段階の目的関数を着せ替えさせ、学習を段階別にすること**。Kくんの見た目（部隊が並ぶ図）に引っ張られず、効いている中身だけ抜く。
