# 自己改善提案 2026-04-10 (Iteration 4)

## 診断サマリー

- **品質スコア**: `data/quality-scores.json` の `scores` は空オブジェクト。採点実績なし。
- **ランク**: 全30エージェントが初期値 `N`。実績なしのため昇格判断不能。
- **使用ログ**: 直近2週間で実運用セッションは事実上ゼロ。記録は初期構築1件＋自己改善ループ3件（Iteration 0/2/3）のみ。Iteration 1 が欠番、Iteration 0 のsummaryが壊れている（`"summary":"自己改善サイクル Iteration 0\n0"` と改行混入）。
- **改善履歴**: improvement-log.jsonl には iteration 1/2/3 が `proposed` で記録されているが、`outputs/improvements/` ディレクトリは空。過去の proposal ファイルが実在しない（未生成 or 消失）。`status` は全て `proposed` のまま、keep/discard 判定に進んだものなし。
- **discarded履歴**: なし。重複回避の制約は今回は適用されない。

**結論**: データは「初回サイクル相当」。根本課題は採点エンジンとロガーが機能していないことで、個別エージェントの改善を論じる前提が未成立。

## 根本原因分析

1. **採点パイプラインが未稼働** — `quality-scores.json` に scores が1件も無い。`monthly-audit.sh` または quality-auditor が実行された形跡がなく、6軸採点が一度も回っていない。改善提案が「どの軸が弱いか」を根拠に作れない。
2. **ロガーの記録粒度不足** — usage-log.jsonl に残るのは session_type/agents_invoked/summary のみで、トークン消費・成果物・参照スキル詳細が空。efficiency 軸と improvement 軸の評価根拠が生成されない。加えて Iteration 0 のJSON行に改行が混入しており、JSONLパーサーが壊れる。
3. **改善ループが proposal を永続化できていない** — `self-improve.sh` が improvement-log.jsonl に行を追加する一方、`outputs/improvements/proposal-*.md` が残っていない。前3イテレーションの内容を参照できず、学習が積み上がらない。ハーネス側の書き込み順序 or 作業ディレクトリの問題が疑われる。

改善ループ自体のハーネスが「書いた気になっているが残らない」状態で、ここを直さない限り iteration N の全てが空回りする。

## 改善提案（優先度順、最大3件）

### 提案1: 自己改善ループの proposal 永続化を担保する [優先度1]

- **対象ファイル**: `scripts/self-improve.sh`
- **変更種別**: スクリプト修正
- **現状の問題**: improvement-log.jsonl は更新されるが `outputs/improvements/proposal-YYYYMMDD.md` が残らない。原因候補は (a) org-designer 呼び出し後に proposal ファイルの存在検証が無い、(b) 一時ディレクトリで実行しコピーし忘れ、(c) そもそもエージェントが Write に失敗しても exit_code 0 で通過。
- **具体的変更**:
  - org-designer 呼び出し直後に `[ -f "outputs/improvements/proposal-$(date +%Y%m%d).md" ] || { echo "proposal missing" >&2; exit 2; }` を追加
  - improvement-log.jsonl への追記は proposal ファイル存在確認 **後** にのみ行う
  - 併せて log 行に `proposal_bytes` フィールドを追加し、0バイト or 欠損を後から検出可能にする
- **期待するスコア影響**: 改善ループ自体の improvement 軸 +5〜8点相当（機能欠損の解消）。副次的に全エージェントの学習材料が蓄積しはじめる。
- **リスク**: 低。検証を追加するだけで既存挙動は壊れない。既に欠損している履歴は復元不可だがこれ以上の欠損を止める。

### 提案2: usage-log.jsonl の壊れ行修正とスキーマ固定 [優先度2]

- **対象ファイル**: `data/usage-log.jsonl`、`scripts/self-improve.sh`（ログ出力部分）、必要なら `CLAUDE.md` の記録仕様
- **変更種別**: データ修正 + スクリプト修正
- **現状の問題**: Iteration 0 行に改行混入で2行に割れており JSONL として不正。スキーマも曖昧でフィールド欠落が常態化。
- **具体的変更**:
  - 壊れた2行目 `0","exit_code":0}` を1行目にマージして1レコードに復元
  - 以降の追記時は `jq -c` を通して書き込むようスクリプトを修正し、改行混入を構造的に防止
  - 必須フィールドを `timestamp, session_type, cost_tier, agents_invoked, skills_referenced, summary, exit_code` に固定
- **期待するスコア影響**: usage-analyst の accuracy +3点、集計系スクリプト全体の信頼性回復。quality-auditor が efficiency 軸を採点する土台になる。
- **リスク**: 低。既存壊れ行の手動マージはレビュー可能。

### 提案3: 初回ベースライン採点の実行トリガー整備 [優先度3]

- **対象ファイル**: `scripts/monthly-audit.sh`（または quality-auditor.md 側の実行条件）
- **変更種別**: スクリプト修正（新規ファイル作成なし）
- **現状の問題**: 採点が一度も走っていないため、改善提案が常に「データ不足」で終わる鶏卵問題。月次トリガーを待たず、初回に限り空 `scores` 検出時に簡易ベースライン採点を走らせる経路が必要。
- **具体的変更**:
  - `monthly-audit.sh` 冒頭で `jq '.scores | length == 0' data/quality-scores.json` が true の場合、quality-auditor を "baseline" モードで 1回呼び出す分岐を追加
  - ベースライン採点は各エージェントに対して accuracy/safety のみ（定義ファイル静的レビュー）で暫定スコアを付け、`scores[agent] = {accuracy, safety, _baseline: true}` として記録
  - 以降の通常採点で `_baseline` フラグを上書きしていく
- **期待するスコア影響**: 全30エージェントに初期スコアが入ることで、次回イテレーションから「最低スコア軸の特定」が実際に機能する。直接のスコア改善ではなく、改善ループ全体の実効性を立ち上げる。
- **リスク**: 中。ベースライン採点の基準が甘いと実運用スコアとの乖離が後で混乱を招くため、`_baseline: true` フラグで区別必須。コスト面はエージェント定義ファイル静的レビューのみなので低予算。

## コスト見積もり

- 提案1〜2: スクリプト修正のみ。追加トークン消費なし（実行時コストむしろ微減）。
- 提案3: 初回に限り quality-auditor を1回起動。静的レビューのため標準ティア（約8,000トークン × 30エージェント分割実行 = 最大240,000トークン相当、1回限り）。以降は通常月次サイクルに吸収されるため継続コスト増なし。
- **月次ランニングコストの増加はゼロ**を維持。

## 前回サイクルの振り返り

- Iteration 1〜3 は improvement-log.jsonl 上では `proposed` と記録されているが、proposal ファイル実体が存在しないため内容検証・keep/discard 判定ができない。
- したがって前3サイクルの学習は事実上ロスト。Iteration 4 では「改善ループの成果物が残ること」を最優先課題に置き直す（提案1）。
- discarded 履歴はゼロのため重複チェックは不要。ただし今回の提案1が通れば、次回以降は前イテレーションの proposal を読んで差分比較できるようになる。

**承認ポイント**: 提案1は即適用可（低リスク・高効果）。提案2はデータ修正を含むため手動レビュー推奨。提案3は採点基準の握りが必要なため、quality-auditor の baseline モード仕様を人間承認してから着手。
