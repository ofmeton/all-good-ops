## 0. 設計の核（人間の役割を最小化、ただし投稿確定だけは人が握る）

チャエン分析で判明した運用本質 = **「発信前工程は重度に自動化、最終投稿は人(iPhone/手動)が握る」**。
これを陸さんの希望フローに落とすと：

```
[自動・常時]  ネタ収集 → 選別 → 下書き文 + メディア素材を作りためる（承認待ちキューに積む）
     ↓
[人・任意]   陸さんが空いた時にキューを見て承認/却下（スマホ可・軽量）
     ↓
[人+Claude]  陸さんがPCを開いたタイミングで、承認済みストックを
            Claude が chrome-devtools で「翌日分の予約投稿」としてXに作成
     ↓
[X]         予約投稿が翌日の最適時間に自動公開（source=本人クライアント維持）
```

ポイント:
- **API直投稿はしない**（チャエンと同じく source=本人クライアントを保ち、誤報リスクに人の最終確認を挟む）
- **予約投稿はX公式の予約機能を chrome-devtools(ログイン済みChrome操作) で叩く** → source問題回避＋手作り感維持
- 陸さんの拘束は「承認（軽い）」＋「PCで予約作成をClaudeに指示（数分）」のみ

## 1. 全体アーキテクチャ

```
┌─ [A] 収集レイヤー（cron/常時, 自動） ──────────────────┐
│ x-buzz-radar (海外Xバズ) + AI企業公式X + PR TIMES/RSS    │
│   ↳ twitterapi.io watchlist 監視 + 重要度/バズ度スコア   │
└──────────────────┬───────────────────────────────────┘
                   ▼
┌─ [B] 生成レイヤー（Claude, 自動） ────────────────────┐
│ 1. 選別: ターゲット(=チャエン層)に刺さるかスコアリング  │
│ 2. 文生成: 固定テンプレに流し込み【速報】下書き         │
│ 3. メディア準備:                                       │
│    - 海外バズ動画/画像 → DL（流用候補, 出典付き）       │
│    - 比較図 → 日本語版を visual-designer/Codex で作成   │
│    - 公式発表 → スクショ/画面録画を自前取得             │
│ 4. 出典リンク確定（自己リプライ用に保持）              │
└──────────────────┬───────────────────────────────────┘
                   ▼
┌─ [C] 承認キュー（Supabase xad schema） ───────────────┐
│ draft: text / media[] / source_url / score / status   │
│ status: pending → approved / rejected                 │
│ 陸さんがWeb or LINE Bot で承認（軽量UI）              │
└──────────────────┬───────────────────────────────────┘
                   ▼
┌─ [D] 投稿レイヤー（人 trigger + Claude, 半自動） ─────┐
│ 陸さんPC起動時:「翌日分を予約作成して」               │
│   ↳ Claude が chrome-devtools でX予約投稿UIを操作      │
│      - approved を時間帯ピークに割当(朝6-8/昼12/夕15-17)│
│      - 本文+メディア添付、出典は予約後スレッドで自己返信 │
└───────────────────────────────────────────────────────┘
```

## 2. 各レイヤー詳細

### [A] 収集レイヤー（既存資産接続）
- **最重要**: all-good-ops に既に `x-buzz-radar`（海外Xバズ収集）がある。チャエンのモデルは実質「x-buzz-radar の収集物を日本語化して再投稿」。**この既存資産を発信側に繋ぐのが最短**
- watchlist 候補（チャエンのネタ元実測より）:
  - 英語AIニュース解説者: `@gerardsans`（チャエン頻出ソース）他
  - AI企業公式: `@AnthropicAI` `@OpenAI` `@GoogleDeepMind` `@NotionDevs` `@ClaudeDevs` 等
  - 海外バズ一般（x-buzz-radar が既にカバー）
- 収集手段: twitterapi.io `user/last_tweets` + `advanced_search`（min_faves でバズ閾値）。コスト $0.15/1000件
- 重要度スコア: いいね/RT速度 + AI企業公式か + 日本語圏での新規性

### [B] 生成レイヤー（Claude）
**固定テンプレ（チャエン実測の黄金型）:**
```
【速報】<主語>が<新機能/出来事>      ← 角括弧hook（速報:朗報≒2:1）
                                     （空行）
<一言の意味づけ/感情>               ← 「強すぎ」「これは必須」「淘汰されそう」
                                     （空行）
・要点1                              ← ・箇条書き3〜5点（保存される設計）
・要点2
・要点3
                                     （空行）
<CTA 👇>                            ← 誘導は5〜6投稿に1回だけ
[メディア添付]                       ← 87%付与原則
```
- 文量: 140〜280字主軸（中央234字）
- メディア準備3経路（親分析§8.6）:
  1. 海外バズ動画/画像 → DL再アップ（**出典をスレッドに明記**、規約リスク低い拡散歓迎系に限定）
  2. 比較図 → 日本語版を作り直し（visual-designer / Codex gpt-image）
  3. 公式発表 → スクショ・画面録画を自前取得
- **著作権/規約ガード**: 基本は引用RT（出典自動担保）。再アップは公式/拡散歓迎本人投稿に限定。compliance エージェントで公開前チェック

### [C] 承認キュー（Supabase xad schema 流用）
```sql
-- xad.post_drafts (案)
id uuid, created_at,
text text,                 -- 生成本文
media jsonb,               -- [{type, local_path, source_url, license_note}]
source_url text,           -- 出典(自己リプライ用)
score numeric,             -- 重要度スコア
category text,             -- 速報/朗報/保存版...
status text,               -- pending|approved|rejected|scheduled|posted
scheduled_for timestamptz, -- 割当時間
approved_at, posted_at, x_tweet_id
```
- 承認UI: 最小は LINE Bot（既に LINE_* 設定あり）or 簡易Web。スマホで「✅/❌」だけ押せる軽量設計
- **陸さんの負荷 = 1日数分の承認のみ**

### [D] 投稿レイヤー（chrome-devtools 予約投稿）
- トリガー: 陸さんがPC起動時に「翌日分を予約作成して」と指示
- Claude の動作:
  1. `status=approved` を取得、翌日の時間帯ピーク（朝6-8/昼12/夕15-17）に割当
  2. chrome-devtools でログイン済みX（`x.com/compose/post`）を操作
  3. 本文入力 → メディア添付（local_path をアップロード）→ **予約日時設定** → 予約確定
  4. 出典がある投稿は「予約投稿への自己リプライ」も予約 or メモ化（スレッドで出典担保）
- **source=本人Chrome/Webクライアント**になり、チャエンの「本人運用感」を再現
- 参考: memory `reference_chrome_devtools_mcp.md`（ログイン済みChrome操作、専用インスタンス方式）

## 3. 投稿リズム（チャエン実測コピー）
- **5〜7投稿/日**、平日厚め・土日薄め
- 時間帯: 朝6-8時 / 昼12時 / 夕15-17時 を主軸
- 構成比: original 67% / 引用27%（うち自己引用で疑似スレッド化）/ RT 5% / **リプライは運用しない（放送型）**

## 4. 段階導入ロードマップ
| Phase | 内容 | 自動化度 |
|---|---|---|
| 0 | テンプレ・時間帯・頻度を手動で模倣運用（型を体で覚える） | 手動 |
| 1 | [A]収集 + [B]文生成のみ自動化、メディアは手動、投稿も手動 | 半自動 |
| 2 | [C]承認キュー + メディア準備自動化 | 半自動 |
| 3 | [D]chrome-devtools 予約投稿を Claude 化（目標フロー完成） | 目標形 |
| 4 | スコアリング学習・自己引用スレッド自動設計 | 最適化 |

## 6. リスク・留意
- **誤報リスク**: AI速報は訂正が起きやすい → [D]の人手承認を必須に（チャエンがAPI自動投稿しない理由）
- **著作権/X規約**: 他者メディア再アップは引用RT基本＋出典明記＋拡散歓迎系限定で低減
- **後発の不利**: チャエン19.4万・専業チームに真っ向勝負は不利（§競合比較参照）。当面は型の完全コピーで運用力を付け、差別化は運用が回ってから
